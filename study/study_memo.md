## 주의사항

#### 요금 
- 인스턴스 스탑

- 중지(Stop)와 종료(Terminate)의 차이:
    - 중지(Stop): 컴퓨터 전원만 끈 상태입니다. CPU 요금은 안 나가지만, 하드디스크(EBS 볼륨)는 남아있어 스토리지 요금이 발생할 수 있습니다.
    - 종료(Terminate): 컴퓨터를 아예 폐기하는 것입니다. 과제를 마치고 리소스를 정리할 때는 반드시 '종료'를 선택해야 합니다.

- 탄력적 IP (Elastic IP, EIP)의 함정: 가장 많이 과금되는 요인 중 하나입니다. EIP는 실행 중인 EC2 인스턴스에 연결되어 있을 때는 무료지만, 인스턴스를 중지(Stop)하거나 삭제했는데 EIP를 반납(Release)하지 않고 쥐고 있으면 시간당 요금이 청구됩니다. 과제를 마치면 반드시 EIP부터 '릴리스(해제)' 하세요.

- 결제 알림(Billing Alarm) 설정: 실습 시작 전 AWS Budgets 서비스나 결제 대시보드에 들어가서 "예상 청구액이 $1를 초과하면 이메일 알림"을 보내도록 가장 먼저 설정하세요.

#### 보안

- 루트(Root) 계정 봉인: 이메일 주소로 로그인하는 루트 계정은 무한대의 권한을 가집니다. 과제 명세서대로 실습용 IAM 사용자를 따로 만들고, 그 사용자에게 필요한 권한(EC2, VPC 등)만 부여하여 로그인(IAM 사용자 로그인) 하세요.

- Key Pair 파일(.pem) 절대 사수: EC2에 접속하기 위해 발급받는 키 페어 파일은 단 한 번만 다운로드할 수 있으며, 비밀번호와 같습니다.
절대 GitHub 같은 퍼블릭 저장소에 코드를 올릴 때 함께 올리지 마세요. 해커들이 봇으로 1분 만에 키를 긁어가서 전 세계 리전에 수백 대의 고성능 서버를 띄워 암호화폐를 채굴합니다(수천만 원 과금 발생).

- 보안 그룹(Security Group) 규칙: 과제 요구사항에 있는 "SSH(22)는 학습자 개인 IP에서만 접근 가능하게 하라"는 필수입니다. 소스(Source)를 내 IP(My IP)로 설정하세요. 0.0.0.0/0(모두 허용)으로 열어두면 전 세계의 해커 봇들이 끊임없이 접속(해킹) 시도를 합니다.

#### 인프라 체크
- 리전(Region) 확인: AWS 콘솔 우측 상단에 지역이 서울(ap-northeast-2) 로 되어 있는지 항상 확인하세요. 나도 모르게 미국(버지니아 북부 등)에 자원을 만들면 나중에 찾지도 못하고 요금만 나옵니다.

- 인터넷 게이트웨이(IGW)와 라우팅 테이블(Route Table): VPC와 서브넷을 만들고 IGW를 붙였다고 끝이 아닙니다. 반드시 서브넷의 '라우팅 테이블'에 가서 0.0.0.0/0 목적지를 만들어 방금 만든 IGW로 향하도록 길을 뚫어주어야 인터넷이 통신됩니다. (초보자가 가장 많이 헤매는 트러블슈팅 포인트 1순위입니다.)

- 퍼블릭 IP 자동 할당: EC2를 생성할 때 네트워크 설정에서 '퍼블릭 IP 자동 할당'이 활성화(Enable) 되어 있어야 외부에서 브라우저로 접속할 수 있는 공인 IP가 부여됩니다.

### 자원 삭제(Cleanup) 순서
- AWS는 만들 때의 역순으로 지워야 깔끔하게 지워집니다. 안 그러면 "의존성 때문에 삭제할 수 없습니다"라는 에러가 뜹니다.
1. EC2 인스턴스 종료(Terminate)
2. (할당했다면) 탄력적 IP(EIP) 연결 해제 후 릴리스(Release)
3. VPC 메뉴로 이동하여 Internet Gateway를 VPC에서 분리(Detach) 후 삭제
4. Subnet 삭제 및 Route Table 삭제
5. 최종적으로 VPC 삭제

## 4. 기능 요구 사항
다음 요구사항을 모두 만족해야 한다.

#### 4.1 네트워크 구성
  - VPC 1개를 생성한다.
  - Public Subnet 1개를 생성한다.
  - Internet Gateway를 VPC에 연결한다.
  - Public Subnet의 Route Table에 0.0.0.0/0 → Internet Gateway 경로가 존재해야 한다.
  - Public Subnet에 배치된 인스턴스는 인터넷으로 아웃바운드 통신이 가능해야 한다. (예: curl <https://example.com> 성공)
#### 4.2 컴퓨트 및 웹 서버 배포
  - Public Subnet에 EC2 인스턴스 1대를 생성한다.
  - SSH로 인스턴스에 접속할 수 있어야 한다.
  - 웹 서버(Nginx 등)를 설치하고 실행 상태여야 한다.
  - 인스턴스에서 curl <http://localhost> 요청이 200 응답을 반환해야 한다.
#### 4.3 접근 제어(보안 그룹)
  - Security Group 인바운드는 원칙적으로 필요한 포트만 허용한다.
  - HTTP(80)는 0.0.0.0/0에서 접근 가능해야 한다.
  - SSH(22)는 학습자 개인 IP(또는 지정된 IP 대역)에서만 접근 가능해야 한다.
  - 0.0.0.0/0에 대해 전체 포트(0-65535) 허용 규칙은 생성하지 않는다.
#### 4.4 권한(IAM) 최소권한
  - 실습에 사용하는 IAM 사용자(또는 Role) 1개를 사용한다.
  - 해당 IAM 권한은 EC2/VPC/Security Group 구성에 필요한 범위로 제한한다.
    - 예: 생성·조회·태그·연결 등 실습에 필요한 작업 중심
    - 예: 실습과 무관한 서비스(S3/RDS 등) 권한은 부여하지 않음
  - 관리자 권한(AdministratorAccess)은 부여하지 않는다.
#### 4.5 외부 접속 검증(택 1)
  - 아래 방식 중 1개를 선택해 “외부에서 접속 가능”을 증명해야 한다.

1. VPC, Subnet, Route Table, Internet Gateway의 역할 및 트래픽 흐름
🌐 핵심 구성 요소의 역할
VPC (Virtual Private Cloud)
역할: AWS 클라우드 내에 정의된 고객만의 논리적으로 격리된 가상 네트워크 공간(영토)입니다. 다른 사용자들로부터 완벽하게 분리되어 독립된 네트워크 환경을 보장합니다.
Subnet (서브넷)
역할: VPC라는 큰 네트워크 영토를 용도나 보안 수준에 따라 더 세분화하여 쪼갠 구역(방)입니다. 인터넷과 양방향 통신이 가능한 **퍼블릭 서브넷(Public Subnet)**과 내부망으로만 쓰이는 **프라이빗 서브넷(Private Subnet)**으로 나뉩니다.
Route Table (라우팅 테이블)
역할: 서브넷 내부에서 발생하는 트래픽이 목적지(IP 대역)를 찾아갈 수 있도록 길을 안내해 주는 이정표(길 안내자)입니다.
Internet Gateway (Internet Gateway, IGW)
역할: VPC와 외부 인터넷 세상을 연결해 주는 유일한 대문(관문)입니다. 사설 IP(Private IP)와 공인 IP(Public IP) 간의 1:1 NAT 변환을 처리하여 통신을 가능케 합니다.
🛣️ 트러블슈팅을 위한 트래픽 흐름 (외부 인터넷 ➔ EC2 인스턴스)
외부 사용자(Client)가 웹 브라우저를 통해 접속을 시도하면 트래픽이 VPC의 관문인 **인터넷 게이트웨이(IGW)**에 도착합니다.
IGW는 유입된 트래픽을 VPC 내부로 전달하며, 해당 서브넷에 연결된 **라우팅 테이블(Route Table)**의 안내를 받습니다.
라우팅 테이블의 규칙(0.0.0.0/0 -> IGW)에 따라 트래픽이 해당 **퍼블릭 서브넷(Public Subnet)**의 목적지 주소로 향합니다.
퍼블릭 서브넷 경계에 진입한 트래픽은 EC2 인스턴스를 감싸고 있는 가상 방화벽인 **보안 그룹(Security Group)**의 인바운드 규칙을 확인합니다.
허용된 포트(예: HTTP 80, HTTPS 443) 규칙을 무사히 통과하면 최종적으로 **EC2 인스턴스 내의 웹 서버(Nginx)**에 도달하여 응답 페이지를 반환합니다.
2. 외부 요청이 EC2의 웹 서버까지 도달하기 위해 필요한 설정
외부 브라우저의 웹 요청이 퍼블릭 서브넷에 배치된 EC2 Nginx 웹 서버까지 정상 도달하려면 다음 3가지 핵심 설정이 유기적으로 맞물려야 합니다.

퍼블릭 IP (Public IP) 할당
논리적 이유: VPC 내부에서 사용하는 IP는 외부에서 접근할 수 없는 사설 IP(Private IP, 예: 10.0.1.x)입니다. 따라서 외부 인터넷 사용자가 브라우저를 통해 통신을 개시할 수 있도록 전 세계에서 고유한 공인 IP 주소(퍼블릭 IP)를 EC2 인스턴스에 반드시 할당해야 합니다.
인터넷 게이트웨이(IGW) 경로 설정 (라우팅)
논리적 이유: 단순히 대문(IGW)을 만드는 것뿐만 아니라, 서브넷에서 외부로 나가는 길을 뚫어주어야 합니다. 퍼블릭 서브넷에 지정된 라우팅 테이블에 **"로컬 VPC 대역을 제외한 모든 외부 인터넷(0.0.0.0/0)으로 향하는 트래픽은 인터넷 게이트웨이(IGW)를 통해라"**는 라우팅 경로가 지정되어 있어야 외부 유입 및 응답 반환 통신이 정상적으로 동작합니다.
보안 그룹 (Security Group) 인바운드 규칙 허용
논리적 이유: AWS의 보안 그룹은 Statefull 방화벽으로, 들어오는 트래픽(인바운드)을 원천 차단하고 허용된 규칙만 통과시킵니다. 따라서 웹 서버로 들어오는 HTTP 서비스 포트인 **80번 포트(및 HTTPS 443번 포트)**에 대해 소스 0.0.0.0/0(모든 IP 대상)으로 허용 규칙이 생성되어 있어야 최종적으로 서버 내부 포트까지 트래픽이 도달할 수 있습니다.
3. 클라우드 과금이 발생하는 대표 요인 및 안전한 리소스 정리 순서
💸 대표적인 과금 요인
EC2 인스턴스 중지(Stop) 상태 방치
인스턴스를 중지하면 컴퓨터 전원만 꺼질 뿐, 저장 장치인 EBS(Elastic Block Store) 볼륨은 그대로 남아있기 때문에 스토리지 보관 요금이 계속 청구됩니다.
미연결 탄력적 IP (Elastic IP, EIP) 방치
탄력적 IP는 한정된 공인 IP 자원이므로, 인스턴스에 연결되어 작동 중일 때는 무료이나 연결된 인스턴스가 없거나 꺼져 있는 상태로 유지되면 점유 시간당 페널티 요금이 과금됩니다.
NAT Gateway, ALB, RDS 등 고단가 리소스 삭제 누락
실습 도중 임시로 생성한 로드 밸런서(ALB)나 데이터베이스(RDS), 프라이빗 통신을 위한 NAT Gateway 등은 시간당 기본 요금이 매우 비싼 편이라 방치 시 큰 금액이 청구됩니다.
🔄 안전한 리소스 정리 순서와 이유
AWS 인프라는 자원 간의 긴밀한 의존 관계(VPC ➔ Subnet ➔ EC2 ➔ Security Group 등)가 얽혀 있습니다. 겉의 네트워크 껍데기(VPC)를 먼저 지우려 하면 안의 내부 자원(EC2) 때문에 의존성 에러가 발생하므로, 반드시 **"안에 들어있는 내용물(EC2)을 먼저 제거한 후 껍데기를 해체하는 역순"**으로 정리해야 합니다.

EC2 인스턴스 종료 (Terminate)
이유: 컴퓨터 본체를 영구 폐기하고, 함께 장착된 EBS 볼륨도 기본 설정에 의해 자동으로 함께 삭제되도록 유도합니다. (상태가 완전히 Terminated 될 때까지 1-2분 대기해야 관련 종속성이 풀립니다.)
탄력적 IP (Elastic IP) 릴리스 (Release)
이유: 인스턴스가 종료되어 연결이 끊어진 상태이므로, EIP를 즉시 반납하여 미연결 IP 자원 점유 과금을 차단합니다.
보안 그룹 (Security Group) 및 키 페어 삭제
이유: EC2 인스턴스에 적용되어 보안 규칙을 담당하던 그룹을 안전하게 폐기합니다.
VPC 삭제
이유: 가장 바깥의 영토인 VPC를 최종적으로 삭제합니다. VPC를 삭제하면 그 안에 속해 있던 서브넷(Subnet), 인터넷 게이트웨이(IGW), 라우팅 테이블(Route Table) 등이 삭제 오류 없이 한 번에 묶여서 안전하게 일괄 자동 삭제됩니다.














## 1. 클라우드와 인프라의 여정: 물리 서버에서 가상화까지

- on-premise: 기업이나 개인이 클라우드 같은 원격 서비스(제3자의 서버)를 빌려 쓰지 않고, 자체적으로 전산실이나 데이터 센터 등 직접 물리적 서버와 IT 인프라를 보유하고 설치하여 운영하는 방식

- hypervisor: 물리 하드웨어 위에 올라가 가상 컴퓨터를 창조하고 통제하는 소프트웨어 레이어.
    - 하드웨어(CPU, Memory, Disk, Network)를 논리적으로 잘게 쪼개어 가상 머신(Virtual Machine, VM)에 나누어 줍니다. VM 내부의 게스트 OS(Guest OS)는 자신이 진짜 물리 컴퓨터 위에서 독점적으로 돌고 있다고 굳게 믿지만, 실제로는 하이퍼바이저가 분할해 준 일부의 자원만을 쓰고 있는 것


- SDN (Software Defined Networking), 네트워크 가상화: 패킷의 헤더에 가상 네트워크 식별자를 붙이고 가상 스위치가 이를 소프트웨어적으로 라우팅 및 캡슐화(Encapsulation)하여 배달하는 방식을 사용.
    - 이러한 고도의 가상 네트워크 기술 위에 구현된, AWS 고객별 독립적인 가상 네트워크 공간이 바로 **VPC (Virtual Private Cloud)**
        - VPC는 쉽게 말해 **"AWS라는 거대한 가상의 우주 안에 나만을 위해 만들어진 독점적이고 고립된 영토"**입니다. 내 VPC 안에서 돌아가는 서버들은 다른 고객의 VPC 내 서버들과 철저히 격리되어, 서로 어떠한 트래픽도 주고받을 수 없는 완벽한 보안 환경을 보장받습니다.

- 서브넷: 이(브로드캐스트 도메인 비대화, 보안) 때문에 우리는 거대한 VPC 영토 안에 **"보안 벽과 파티션(라우팅 경계)"**을 쳐서 네트워크 영역을 쪼개게 되는데, 이 쪼개진 방들을 **서브넷 (Subnet, 하위 네트워크)**이라고 부릅니다.

- 퍼블릭 서브넷
- AWS 서브넷에서 사라진 5개의 예약 IP 주소

- 로컬 라우팅(Local Route) 규칙: VPC 내부에서 서브넷을 만들면, 그 서브넷의 라우팅 테이블에는 아무 짓도 하지 않아도 무조건 가장 먼저 등록되어 있고 절대 삭제할 수 없는 절대적인 규칙

- 세상과 소통하는 관문, Internet Gateway (IGW)
    - 외부 통로 제공: VPC 내부의 자원들이 바깥 인터넷에 접속하고, 바깥 인터넷 사용자들도 VPC 내부로 유입될 수 있는 관문이 됩니다.
    - 1:1 Static NAT (Network Address Translation) 처리: 이것이 바로 IGW의 핵심 마법입니다.
    - WS의 인터넷 게이트웨이는 이름만 '게이트웨이'일 뿐, 창고에 서 있는 단일 물리 장비나 가상 머신이 아닙니다. 사실 IGW는 AWS 소프트웨어 정의 네트워크(SDN) 상에 널리 분산되어 논리적으로 동작하는 고가용성, 초확장성 가상 커넥터입니다.
    - 실제 트래픽은 데이터 센터 내의 수많은 물리 스위치 노드들을 통해 고도로 분산되어 처리되므로, 대역폭의 한계가 없고 하드웨어 결함으로 인해 IGW가 죽는 일 또한 구조적으로 일어나지 않습니다. AWS가 우리에게 제공하는 완벽한 가상화 추상화의 이점입니다.

- EBS (Elastic Block Store): 가상 컴퓨터의 하드디스크

- Nginx의 구조적 정의
    - Nginx는 비동기 이벤트 기반(Event-Driven) 구조를 가진 웹 서버 소프트웨어입니다. 클라이언트(브라우저)의 요청을 받아 정적 콘텐츠를 응답하거나, 뒤에 있는 애플리케이션 서버로 요청을 전달하는 역할을 수행합니다.
    - 구조적 위치: 사용자(Client)와 백엔드 서버(Application Server, 예: Node.js, Python, Java) 사이의 **'앞단'**에 위치하여 게이트웨이 역할을 합니다.
    - 작동 방식: 과거의 웹 서버(Apache 등)가 요청 하나당 프로세스나 스레드를 하나씩 할당하여 자원을 많이 소모했던 것과 달리, Nginx는 적은 수의 프로세스로 수만 개의 연결을 동시에 처리하는 비동기 방식을 채택하고 있습니다.

### 1. VPC, Subnet, Route Table, Internet Gateway의 역할 및 트래픽 흐름

#### 🌐 핵심 구성 요소의 역할
* **VPC (Virtual Private Cloud)**
  * **역할**: AWS 클라우드 내에 정의된 고객만의 논리적으로 격리된 가상 네트워크 공간(영토)입니다. 다른 사용자들로부터 완벽하게 분리되어 독립된 네트워크 환경을 보장합니다.
* **Subnet (서브넷)**
  * **역할**: VPC라는 큰 네트워크 영토를 용도나 보안 수준에 따라 더 세분화하여 쪼갠 구역(방)입니다. 인터넷과 양방향 통신이 가능한 **퍼블릭 서브넷(Public Subnet)**과 내부망으로만 쓰이는 **프라이빗 서브넷(Private Subnet)**으로 나뉩니다.
* **Route Table (라우팅 테이블)**
  * **역할**: 서브넷 내부에서 발생하는 트래픽이 목적지(IP 대역)를 찾아갈 수 있도록 길을 안내해 주는 이정표(길 안내자)입니다.
* **Internet Gateway (Internet Gateway, IGW)**
  * **역할**: VPC와 외부 인터넷 세상을 연결해 주는 유일한 대문(관문)입니다. 사설 IP(Private IP)와 공인 IP(Public IP) 간의 1:1 NAT 변환을 처리하여 통신을 가능케 합니다.

#### 🛣️ 트러블슈팅을 위한 트래픽 흐름 (외부 인터넷 ➔ EC2 인스턴스)
1. 외부 사용자(Client)가 웹 브라우저를 통해 접속을 시도하면 트래픽이 VPC의 관문인 **인터넷 게이트웨이(IGW)**에 도착합니다.
2. IGW는 유입된 트래픽을 VPC 내부로 전달하며, 해당 서브넷에 연결된 **라우팅 테이블(Route Table)**의 안내를 받습니다.
3. 라우팅 테이블의 규칙(`0.0.0.0/0 -> IGW`)에 따라 트래픽이 해당 **퍼블릭 서브넷(Public Subnet)**의 목적지 주소로 향합니다.
4. 퍼블릭 서브넷 경계에 진입한 트래픽은 EC2 인스턴스를 감싸고 있는 가상 방화벽인 **보안 그룹(Security Group)**의 인바운드 규칙을 확인합니다.
5. 허용된 포트(예: HTTP 80, HTTPS 443) 규칙을 무사히 통과하면 최종적으로 **EC2 인스턴스 내의 웹 서버(Nginx)**에 도달하여 응답 페이지를 반환합니다.

### 2. 외부 요청이 EC2의 웹 서버까지 도달하기 위해 필요한 설정

외부 브라우저의 웹 요청이 퍼블릭 서브넷에 배치된 EC2 Nginx 웹 서버까지 정상 도달하려면 다음 3가지 핵심 설정이 유기적으로 맞물려야 합니다.

* **퍼블릭 IP (Public IP) 할당**
  * **논리적 이유**: VPC 내부에서 사용하는 IP는 외부에서 접근할 수 없는 사설 IP(Private IP, 예: `10.0.1.x`)입니다. 따라서 외부 인터넷 사용자가 브라우저를 통해 통신을 개시할 수 있도록 전 세계에서 고유한 공인 IP 주소(퍼블릭 IP)를 EC2 인스턴스에 반드시 할당해야 합니다.
* **인터넷 게이트웨이(IGW) 경로 설정 (라우팅)**
  * **논리적 이유**: 단순히 대문(IGW)을 만드는 것뿐만 아니라, 서브넷에서 외부로 나가는 길을 뚫어주어야 합니다. 퍼블릭 서브넷에 지정된 라우팅 테이블에 **"로컬 VPC 대역을 제외한 모든 외부 인터넷(`0.0.0.0/0`)으로 향하는 트래픽은 인터넷 게이트웨이(`IGW`)를 통해라"**는 라우팅 경로가 지정되어 있어야 외부 유입 및 응답 반환 통신이 정상적으로 동작합니다.
* **보안 그룹 (Security Group) 인바운드 규칙 허용**
  * **논리적 이유**: AWS의 보안 그룹은 Statefull 방화벽으로, 들어오는 트래픽(인바운드)을 원천 차단하고 허용된 규칙만 통과시킵니다. 따라서 웹 서버로 들어오는 HTTP 서비스 포트인 **80번 포트(및 HTTPS 443번 포트)**에 대해 소스 `0.0.0.0/0`(모든 IP 대상)으로 허용 규칙이 생성되어 있어야 최종적으로 서버 내부 포트까지 트래픽이 도달할 수 있습니다.

---

### 3. 클라우드 과금이 발생하는 대표 요인 및 안전한 리소스 정리 순서

#### 💸 대표적인 과금 요인
1. **EC2 인스턴스 중지(Stop) 상태 방치**
   * 인스턴스를 중지하면 컴퓨터 전원만 꺼질 뿐, 저장 장치인 **EBS(Elastic Block Store) 볼륨**은 그대로 남아있기 때문에 스토리지 보관 요금이 계속 청구됩니다.
2. **미연결 탄력적 IP (Elastic IP, EIP) 방치**
   * 탄력적 IP는 한정된 공인 IP 자원이므로, 인스턴스에 연결되어 작동 중일 때는 무료이나 **연결된 인스턴스가 없거나 꺼져 있는 상태로 유지되면 점유 시간당 페널티 요금이 과금**됩니다.
3. **NAT Gateway, ALB, RDS 등 고단가 리소스 삭제 누락**
   * 실습 도중 임시로 생성한 로드 밸런서(ALB)나 데이터베이스(RDS), 프라이빗 통신을 위한 NAT Gateway 등은 시간당 기본 요금이 매우 비싼 편이라 방치 시 큰 금액이 청구됩니다.

#### 🔄 안전한 리소스 정리 순서와 이유
AWS 인프라는 자원 간의 긴밀한 의존 관계(VPC ➔ Subnet ➔ EC2 ➔ Security Group 등)가 얽혀 있습니다. 겉의 네트워크 껍데기(VPC)를 먼저 지우려 하면 안의 내부 자원(EC2) 때문에 의존성 에러가 발생하므로, 반드시 **"안에 들어있는 내용물(EC2)을 먼저 제거한 후 껍데기를 해체하는 역순"**으로 정리해야 합니다.

1. **EC2 인스턴스 종료 (Terminate)**
   * **이유**: 컴퓨터 본체를 영구 폐기하고, 함께 장착된 EBS 볼륨도 기본 설정에 의해 자동으로 함께 삭제되도록 유도합니다. (상태가 완전히 `Terminated` 될 때까지 1-2분 대기해야 관련 종속성이 풀립니다.)
2. **탄력적 IP (Elastic IP) 릴리스 (Release)**
   * **이유**: 인스턴스가 종료되어 연결이 끊어진 상태이므로, EIP를 즉시 반납하여 미연결 IP 자원 점유 과금을 차단합니다.
3. **보안 그룹 (Security Group) 및 키 페어 삭제**
   * **이유**: EC2 인스턴스에 적용되어 보안 규칙을 담당하던 그룹을 안전하게 폐기합니다.
4. **VPC 삭제**
   * **이유**: 가장 바깥의 영토인 VPC를 최종적으로 삭제합니다. VPC를 삭제하면 그 안에 속해 있던 **서브넷(Subnet), 인터넷 게이트웨이(IGW), 라우팅 테이블(Route Table)** 등이 삭제 오류 없이 한 번에 묶여서 안전하게 일괄 자동 삭제됩니다.

**라우팅 테이블(Route Table)**과 **보안 그룹(Security Group)**은 둘 다 AWS 네트워크에서 트래픽을 통제하는 핵심 요소이지만, 역할과 작동 방식에서 근본적인 차이가 있습니다.

가장 쉽게 비유하자면, 라우팅 테이블은 **도로 위의 '이정표(네비게이션)'**이고, 보안 그룹은 **건물 입구의 '보안 경비원(가상 방화벽)'**입니다.

---

### 1. 라우팅 테이블 (Route Table) : "길 안내자"
* **역할**: 트래픽이 목적지(IP 대역)에 도달하기 위해 **어떤 통로(인터넷 게이트웨이, NAT 게이트웨이 등)를 거쳐서 가야 하는지 경로**를 지정합니다.
* **작동 범위**: **서브넷(Subnet) 레벨**에서 작동합니다. 서브넷 내의 모든 리소스에서 나가는 트래픽에 일괄 적용됩니다.
* **주요 특징**:
  * 패킷의 차단/허용을 결정하기보다는 **"A 목적지로 가려면 B 문으로 나가라"**고 중계해 줍니다.
  * 예: 외부 인터넷(`0.0.0.0/0`)으로 나가는 길은 인터넷 게이트웨이(`igw-xxx`)로 가라고 지정하는 규칙.

### 2. 보안 그룹 (Security Group) : "방화벽"
* **역할**: 인스턴스로 들어오거나 나가는 트래픽을 **프로토콜, 포트, 소스 IP 주소**를 기준으로 검사하여 **허용할지 차단할지** 결정합니다.
* **작동 범위**: **인스턴스(Network Interface/ENI) 레벨**에서 개별적으로 작동합니다.
* **주요 특징**:
  * **허용(Allow) 규칙**만 설정할 수 있으며, 여기에 명시되지 않은 모든 트래픽은 기본적으로 차단됩니다.
  * **상태 저장(Stateful) 방식**으로 동작합니다. 즉, 인바운드(들어오는 길)가 허용되면 그에 대한 아웃바운드(나가는 응답) 트래픽은 아웃바운드 규칙과 상관없이 자동으로 허용됩니다.

### 💡 흐름 예시
사용자가 브라우저에 주소를 쳐서 우리 서버로 올 때:
1. **라우팅 테이블**: "음, `10.0.1.5`인 EC2가 있는 Public Subnet으로 가고 있군. 이쪽 길(Subnet)로 안내해 줄게."
2. **보안 그룹**: "잠깐! 너 포트 80(HTTP)으로 들어왔고 외부 사용자가 맞네. 규칙에 허용되어 있으니 EC2 서버 안으로 들여보내 줄게."

사용자가 `http://`로 접속했을 때 `https://`로 자동으로 전환(리다이렉션)되는 과정은 **웹 서버(Nginx)**와 **웹 브라우저(크롬, 사파리 등)** 간의 주고받는 약속에 의해 이루어집니다. 

작성하신 Nginx 설정 파일(`default.conf`)의 코드를 기준으로 그 작동 원리를 단계별로 설명해 드릴게요.

---

### 1. 설정 코드 다시 보기

Nginx 설정 파일에는 두 개의 `server` 블록이 존재합니다.

```nginx
# 1번 블록: HTTP (80번 포트) 통제소
server {
    listen 80;
    server_name [도메인];

    # HTTP 요청을 HTTPS로 301 강제 리다이렉트
    return 301 https://$host$request_uri;
}

# 2번 블록: HTTPS (443번 포트) 통제소
server {
    listen 443 ssl;
    server_name [도메인];
    
    # ... SSL 인증서 설정 및 웹페이지 서빙 ...
}
```

---

### 2. 리다이렉팅이 일어나는 4단계 과정

#### **1단계: 사용자의 HTTP 접속 시도**
* 사용자가 브라우저 주소창에 `http://[도메인]`으로 입력하고 접속하면, 브라우저는 기본적으로 **HTTP의 기본 포트인 80번 포트**를 통해 서버에 접속을 시도합니다.

#### **2단계: Nginx의 80번 포트 서버 블록 작동**
* Nginx는 80번 포트로 들어오는 요청을 받고 **1번 블록(listen 80)**을 실행합니다.
* 1번 블록의 내부 명령어인 `return 301 https://$host$request_uri;`가 실행됩니다.
  * **`301`**: 브라우저에게 *"이 주소는 영구적으로 이동(Moved Permanently)했어"*라고 알려주는 HTTP 상태 코드입니다.
  * **`https://$host$request_uri`**: 원래 요청했던 호스트 이름(도메인)과 상세 주소(URI)는 그대로 유지하되, 맨 앞의 프로토콜만 `https://`로 바꾸어 새로운 주소를 만듭니다.

#### **3단계: 브라우저의 리다이렉션 요청 재시도**
* Nginx 서버로부터 **301 상태 코드**와 함께 **"새로운 주소: `https://[도메인]`"**이라는 응답을 받은 웹 브라우저는 사용자가 다시 입력하지 않아도 **자동으로** 해당 HTTPS 주소로 다시 요청을 보냅니다.
* 이때 프로토콜이 `https`로 바뀌었기 때문에, 브라우저는 **HTTPS의 기본 포트인 443번 포트**로 서버에 다시 문을 두드립니다.

#### **4단계: Nginx의 443번 포트 서버 블록 작동 및 페이지 표시**
* Nginx는 443번 포트로 들어온 재요청을 받고 **2번 블록(listen 443 ssl)**을 작동시킵니다.
* SSL 인증서를 통해 브라우저와 암호화 통신(TLS Handshake)을 맺은 뒤, 비로소 Nginx의 웰컴 페이지를 사용자 브라우저에 안전하게(HTTPS) 띄워줍니다.

---

### 💡 요약
사용자가 HTTP(80)로 접속하면, **Nginx**가 *"거기는 보안이 안 되니 HTTPS(443) 주소로 다시 가라"*고 **301 응답**을 주고, **브라우저**가 이 응답을 해석해서 **알아서 HTTPS 주소로 다시 접속**하는 것입니다.

---

트래픽이 증가하여 인스턴스를 2대 이상으로 늘리는 상황(스케일 아웃, Scale-Out)에서 **현재 구조의 병목 요인**과 **이를 해결하기 위해 추가해야 할 구성 요소**에 대한 설명입니다.

---

### 1. 현재 구조의 병목 및 문제점 (Bottleneck)

* **단일 장애점 (SPOF, Single Point of Failure)**
  * 현재는 단 1대의 EC2 인스턴스가 서비스를 처리하므로, 서버에 장애가 발생하면 전체 서비스가 완전히 중단됩니다.
* **트래픽 분산(부하 분산) 장치 부재**
  * 외부 사용자는 현재 단일 EC2의 퍼블릭 IP(또는 이에 연결된 도메인)로 직접 접속합니다. 인스턴스를 2대로 늘리더라도 트래픽을 두 서버로 나누어 보내주는 장치가 없습니다.
* **SSL/TLS 인증서 관리의 한계**
  * 현재는 단일 EC2 내부에 Let's Encrypt 인증서를 보관하고 웹 서버가 SSL 암호화/복호화를 처리합니다. 서버가 2대로 늘어나면 각 서버에 인증서를 복사해야 하고 동기화 및 갱신 관리가 매우 까다로워집니다.
* **단일 가용 영역(AZ) 의존성**
  * 현재 VPC 네트워크 구조는 1개의 서브넷(ap-northeast-2a)에만 치중되어 있습니다. 만약 해당 AWS 데이터 센터 영역(가용 영역) 전체에 장애가 발생하면 대처할 수 없습니다.

---

### 2. 해결을 위해 추가해야 할 핵심 구성 요소

이 문제를 해결하고 고가용성(High Availability) 및 부하 분산을 달성하기 위해 다음 구성 요소를 도입해야 합니다.

```
                  [ Internet ]
                       │
             [ Internet Gateway ]
                       │
          [ Application Load Balancer ] (ACM SSL 적용)
             ┌─────────┴─────────┐
             ▼                   ▼
     [ Public Subnet A ]   [ Public Subnet B ] (가용 영역 분산)
      ┌─────────────┐       ┌─────────────┐
      │   EC2-01    │       │   EC2-02    │
      └─────────────┘       └─────────────┘
```

#### ① **ALB (Application Load Balancer, 애플리케이션 로드 밸런서)**
* **역할**: 외부 사용자의 요청을 수신하는 단일 창구(엔드포인트) 역할을 하며, 들어오는 트래픽을 뒤에 있는 2대의 EC2 인스턴스로 균등하게 나누어 보냅니다(부하 분산).
* **상태 검사 (Health Check)**: 주기적으로 EC2 인스턴스들의 상태를 감시하여, 한 대가 죽으면 자동으로 정상적인 다른 인스턴스로만 트래픽을 보냅니다.
* **SSL/TLS Offloading (인증서 오프로딩)**: AWS Certificate Manager(ACM)를 활용해 **SSL 인증서를 ALB에 적용**합니다. 외부 사용자와 ALB 사이는 HTTPS(443) 보안 통신을 하고, ALB와 뒤편의 EC2 간에는 일반 HTTP(80) 통신을 수행하여 EC2의 암호화 연산 부담을 덜고 인증서 관리를 단일화합니다.

#### ② **추가 가용 영역(AZ) 및 서브넷 (Multi-AZ 구성)**
* **역할**: 고가용성을 보장하기 위해 기존 서브넷 외에 다른 가용 영역(예: `ap-northeast-2c`)에 **두 번째 Public Subnet을 생성**하고, 두 번째 EC2 인스턴스는 이곳에 배치합니다. 
* **이유**: ALB는 고가용성을 위해 최소 2개 이상의 가용 영역에 걸쳐 있는 서브넷들을 요구하기 때문입니다.

#### ③ **대상 그룹 (Target Group)**
* **역할**: ALB가 부하를 분산할 대상(EC2 인스턴스 2대)을 논리적으로 묶어주는 논리적인 그룹입니다.

#### ④ **Auto Scaling Group (ASG - 선택이지만 필수 권장)**
* **역할**: 트래픽 증가에 따라 인스턴스를 자동으로 2대, 3대로 늘려주고(Scale-Out), 새벽 시간대처럼 한산할 때는 자동으로 줄여주는(Scale-In) 유연한 자원 관리를 처리합니다.

---

### 3. 개선 후의 트래픽 흐름

1. 사용자는 도메인(`mission61.ddns.net`)을 호출하지만, 이 도메인은 이제 개별 EC2의 IP가 아니라 **ALB의 도메인(DNS Name)**을 가리킵니다.
2. 트래픽이 **인터넷 게이트웨이**를 거쳐 **ALB**에 도달합니다.
3. ALB는 요청의 SSL 암호화를 해제(복호화)한 뒤, 라운드 로빈 등의 알고리즘에 따라 **가장 여유 있는 서브넷의 EC2 인스턴스**로 일반 HTTP(80) 패킷을 보냅니다.
4. 만약 인스턴스 1대에 장애가 나더라도 ALB가 즉시 감지하여 정상적인 다른 인스턴스 하나로만 서비스가 끊김 없이 유지되도록 처리합니다.

---

AWS 결제 대시보드(Billing Dashboard)에 예상치 못한 비용이 발생했을 때, **가장 먼저 의심해야 할 리소스**와 **이를 추적하고 확실하게 정리하는 단계적 절차**입니다.

---

### 1. 어떤 리소스부터 의심해야 하는가? (의심 순위)

AWS 실습 중 예상치 못한 과금이 발생했을 때 대부분 다음 리소스가 원인입니다.

* **1순위: NAT Gateway (NAT 게이트웨이)**
  * **과금 이유**: 프라이빗 서브넷의 아웃바운드 통신을 위해 생성하는 NAT Gateway는 **실행 시간당 요금(서울 리전 기준 약 $0.059/시간)**이 매우 비싸게 책정됩니다. 단지 켜놓기만 해도 한 달에 약 **$43(약 6만 원) 이상**의 청구 요금이 고정적으로 발생합니다.
* **2순위: 미연결 탄력적 IP (Elastic IP, EIP)**
  * **과금 이유**: 사용 가능한 IPv4 주소 자원의 고갈로 인해, EC2 인스턴스가 꺼져 있거나 연결되지 않은 EIP는 시간당 패널티 요금(약 $0.005/시간)이 과금됩니다.
* **3순위: 미사용 EBS 볼륨 (EBS Orphaned Volumes)**
  * **과금 이유**: EC2 인스턴스를 종료(Terminate)할 때 '종료 시 삭제' 옵션을 해제했거나, 인스턴스는 지웠지만 하드디스크 역할을 하는 EBS 볼륨은 지워지지 않아 `Available(미사용)` 상태로 방치되어 스토리지 요금이 나가는 경우입니다.
* **4순위: 타 리전(Region)에 방치된 리소스**
  * **과금 이유**: AWS 콘솔 화면 우상단 리전이 버지니아 북부, 오레곤 등으로 지정된 상태에서 나도 모르게 자원(EC2 등)을 생성한 후, 서울 리전 콘솔만 확인하고 "다 지웠는데 돈이 나간다"고 착각하는 경우입니다.
* **5순위: ELB (로드 밸런서) 및 RDS (데이터베이스)**
  * **과금 이유**: 로드 밸런서(ALB) 및 RDS 인스턴스는 프리티어 기본 범위를 넘어서거나 실습 종료 후 삭제를 누락했을 때 고정 비용이 높게 발생하는 단골 리소스입니다.

### 2. 비용을 어떻게 추적할 것인가? (추적 방법)

#### ① **Billing Dashboard ➔ Bills (청구서) 메뉴 확인**
* 비용이 발생한 **연도/월**을 선택하고 청구 항목을 확장하여 확인합니다.
* **"어떤 서비스(Service)"**에서 비용이 나갔는지, 그리고 **"어떤 리전(Region)"**에서 발생했는지 아주 구체적으로 적혀 있습니다. (예: *Virtual Private Cloud - APN2-NatGateway-Hours*)

#### ② **AWS Cost Explorer (비용 탐색기) 활용**
* 서비스별, 리전별 일일 비용 변화 추이를 시각적 그래프로 볼 수 있어 급격히 비용이 튄 날짜와 서비스를 직관적으로 탐색할 수 있습니다.

#### ③ **AWS Resource Explorer (리소스 탐색기) 활용**
* 상단 검색창에 `Resource Explorer`를 검색하여 사용합니다.
* 전 세계 모든 AWS 리전(Region)에 흩어져 있는 나의 자원들을 검색 필터(`service:ec2` 등)를 통해 **일괄 검색**할 수 있어, 타 리전에 깜빡하고 두고 온 자원을 찾아내는 데 매우 효과적입니다.

---

### 3. 어떻게 정리할 것인가? (정리 절차)

추적을 통해 원인 요소를 파악했다면 아래 순서에 맞추어 완전 정리를 진행합니다.

1. **해당 리전으로 전환**: 비용이 발생한 리전(예: 미국 동부 버지니아 북부 등)으로 AWS 콘솔 우상단 리전을 변경합니다.
2. **핵심 원인 자원 해제 및 종료**:
   * **NAT Gateway**: [VPC] ➔ [NAT 게이트웨이] 메뉴에서 즉시 '삭제(Delete)' 합니다.
   * **ELB / RDS**: 사용하지 않는 로드 밸런서와 데이터베이스 인스턴스를 삭제합니다. (RDS 삭제 시 최종 스냅샷 생성은 불필요하다면 체크 해제)
   * **EBS 볼륨**: [EC2] ➔ [볼륨] 메뉴에서 상태가 `in-use`가 아닌 **`available` 상태인 볼륨을 선택해 '삭제(Delete)'** 합니다.
   * **탄력적 IP**: [EC2] ➔ [탄력적 IP] 메뉴에서 주소를 선택하고 **'탄력적 IP 주소 릴리스(Release)'**를 실행해 완전히 반납합니다.
3. **Billing Dashboard에서 24시간 뒤 재확인**: AWS의 요금 데이터는 실시간 업데이트에 수 시간에서 최대 24시간이 소요되므로, 정리를 마친 다음 날 일일 요금 그래프가 0으로 내려앉았는지 최종 점검합니다.

#### 💡 예방 팁
실습 계정 생성 직후 **AWS Budgets**에 접속하여 *"월 예상 청구 금액이 $1(또는 $5)를 초과할 경우 내 이메일로 경보 알림 발송"* 예산을 가장 먼저 설정해 두는 것이 과금 폭탄을 막는 최선의 예방책입니다.

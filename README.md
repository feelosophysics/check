# AI 기반 Git 커밋 & PR 자동 생성기

이 프로젝트는 Git 리포지토리의 변경 사항(`git diff`, `git status`)을 감지하고, Google Gemini AI API를 활용하여 규칙과 컨벤션에 맞는 커밋 메시지와 Pull Request(PR) 초안을 자동으로 생성하는 CLI 도구입니다.

## 주요 기능
- **커밋 메시지 생성**: 변경 사항을 요약하여 제목과 본문을 포함한 커밋 메시지를 생성합니다.
- **PR 초안 생성**: Why, What, How to Test 구조에 맞춘 PR 템플릿을 생성합니다.
- **팀 컨벤션 적용**: `.ai-gitgen.yml` 설정 파일을 통해 커스텀 커밋 프리픽스와 PR 요구사항을 쉽게 적용할 수 있습니다.
- **안전 모드 (Safe Mode)**: 민감한 정보(이메일, API 키 등)를 마스킹 처리하고, 전송되는 줄 수를 제한하여 토큰 낭비와 정보 유출을 방지합니다.

## 🚀 설치 및 환경 설정

### 1. 파이썬 라이브러리 설치
이 도구는 기본 파이썬 라이브러리를 주로 사용하지만, `.env` 파일 로드 및 컨벤션 설정(`.ai-gitgen.yml`)을 지원하기 위해 `python-dotenv` 및 `pyyaml` 라이브러리가 필요합니다.
```bash
pip install python-dotenv pyyaml
```

### 2. API Key 설정 (필수)
Google Gemini AI를 사용하기 위해서는 API Key 발급 및 등록이 필요합니다. **보안을 위해 `.env` 파일 방식을 강력히 권장합니다.**

1. [Google AI Studio](https://aistudio.google.com/)에 접속하여 API 키를 발급받습니다.
2. **방법 A: `.env` 파일 사용 (추천 - 보안 강화)**
   - 프로젝트 루트 디렉토리에 있는 `.env.example` 파일을 복사하여 `.env` 파일을 생성합니다.
     ```bash
     cp .env.example .env
     ```
   - `.env` 파일을 텍스트 에디터로 열어 실제 발급받은 API 키를 입력합니다.
     ```env
     AI_API_KEY=여러분의_발급된_API_키
     ```
   - 이 프로젝트의 `.gitignore`에는 이미 `.env`가 등록되어 있으므로, 실제 키가 포함된 파일이 깃허브 등에 유출되지 않습니다.
3. **방법 B: 터미널 세션 환경변수 직접 등록 (임시 사용)**
   - 파일을 만들지 않고 터미널 세션 동안에만 임시로 등록해 사용하려면 아래 명령을 사용합니다.
     - **Mac/Linux:** `export AI_API_KEY="여러분의_발급된_API_키"`
     - **Windows(CMD):** `set AI_API_KEY="여러분의_발급된_API_키"`
     - **Windows(PowerShell):** `$env:AI_API_KEY="여러분의_발급된_API_키"`

## 💡 아키텍처 및 설계 결정 (평가 기준 답변)

프로젝트 구현 과정에서 반영된 주요 기술적/설계적 결정 사항들입니다.

### 1. Git 로직과 AI API 연동 로직의 분리
**이유**: 단일 책임 원칙(SRP)을 지키고, 유지보수성을 높이기 위함입니다. Git 변경 사항을 수집하는 로직(`get_git_status`, `get_git_diff`)과 이를 바탕으로 AI API를 호출하는 로직(`call_gemini_api`)을 분리함으로써, 추후 Git 대신 다른 버전 관리 도구를 사용하거나 다른 AI 모델(예: OpenAI)로 교체할 때 영향을 최소화할 수 있습니다.

### 2. 프롬프트 구성과 출력 포맷팅의 분리
**이유**: 코드 상에서 프롬프트 템플릿에 Git Diff 데이터와 요구사항 규칙을 주입하는 역할과, API의 순수 호출 및 JSON 응답 파싱 역할을 분리했습니다. 이를 통해 상황(Commit vs PR)에 맞춰 프롬프트를 유연하게 조립할 수 있습니다.

### 3. API 파라미터의 CLI 옵션화
**이유**: AI의 응답은 비결정적(Non-deterministic)이므로, 개발자가 최적의 결과를 찾기 위해서는 다양한 실험이 필요합니다. `temperature`, `max_tokens` 등을 CLI 옵션(`argparse` 활용)으로 열어두어 코드를 직접 수정하지 않고도 신속하게 결과 차이를 테스트하고 재현할 수 있게 설계했습니다.

### 4. 오류 및 예외 처리 구현
**방식**: `urllib`의 예외 처리(`HTTPError`, `URLError`)를 세분화하여, 404(모델 경로 없음), 429(할당량 초과) 등 구체적인 원인과 해결책을 사용자에게 터미널에 안내하도록 구현했습니다. 또한 API Key 누락 시 즉시 실행을 중단하도록 검증 로직을 추가했습니다.
**이유**: CLI 도구의 특성상 문제가 발생했을 때 터미널 출력만으로 사용자가 빠르게 원인을 파악하고 스스로 대처할 수 있도록 돕기 위해서입니다.

### 5. 파라미터 조작에 따른 결과의 차이
- **Temperature**: 값이 낮을수록(0.0에 가까움) 일관되고 보수적인 어휘를 선택하여 사실 위주의 요약을 생성하며, 높을수록(1.0에 가까움) 더 창의적이고 다양한 문맥을 담은 문장 구조를 보여줍니다.
- **Max Tokens**: 이 값은 AI가 생성할 수 있는 텍스트의 최대 길이를 제한합니다. 너무 낮게 설정하면 답변이 중간에 잘리게 되며, 적절히 제한(예: 500)해두면 무한 루프에 빠지거나 불필요한 토큰 비용이 낭비되는 것을 막아주는 역할을 합니다.

### 6. 출력 길이/형식 제어 전략 (재생성 vs 후처리)
**선택**: 저희는 코드 레벨의 '후처리(자르기)'가 아닌 **'프롬프트 내 지시(재생성 유도)'** 방식을 선택했습니다. (예: "50자 이내로 작성해주세요")
**이유**: 문자열 기반의 기계적인 후처리를 하면 텍스트의 문맥이 잘리거나 어색해질 수 있습니다. AI가 애초에 문맥을 유지하면서 스스로 길이를 맞추도록 유도하는 것이 자연스러운 커밋 메시지와 PR을 얻는 데 더 효과적입니다.

### 7. AI 생성 결과물 검토의 필요성
**이유**: AI는 완벽하지 않으며 환각(Hallucination) 현상으로 인해 코드에 없는 내용을 지어내거나 중요한 변경 사항을 누락할 가능성이 존재합니다. 따라서 도구의 출력물은 "초안(Draft)"으로 간주하고, 개발자가 직접 리뷰한 뒤 최종적으로 반영해야 합니다.

### 8. 향후 개선 방향 (팀 프로젝트 적용 시 우선순위)
**1순위**: PR 생성 후 자동으로 GitHub에 업로드(연동)까지 해주는 기능
**근거**: 현재는 터미널에 출력된 내용을 복사-붙여넣기 해야 하는 번거로움이 있습니다. 자동화의 실질적인 효율을 극대화하기 위해, Github CLI나 API와 연동하여 자동으로 PR 티켓을 생성해 주는 파이프라인을 구축하는 것이 가장 임팩트가 클 것이라고 판단했습니다.

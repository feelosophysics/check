# 🗄️ 냉장고 관리 시스템 데이터베이스 피어 리뷰 (Peer Review) Report

이 프로젝트는 SQLite를 사용하여 셰어하우스 냉장고 관리 도메인을 매우 체계적으로 설계하고, 필수 요구사항인 스키마 정의, 샘플 데이터 입력, 15개의 핵심 쿼리 작성을 훌륭하게 완수했습니다. 특히 **보너스 실습**에서 SQLite의 외래 키 제한(`PRAGMA foreign_keys = ON;`) 특성을 직접 검증하고, 비즈니스 핵심 지표 3개를 정의하여 훌륭한 쿼리 실습을 보여준 점이 매우 인상적입니다.

본 리뷰 문서에서는 프로젝트의 완성도를 한 단계 더 높이기 위해 개선할 수 있는 **논리적 취약점, 데이터 정합성 이슈, 성능 최적화 포인트**를 정리했습니다.

---

## 🚨 1. 치명적인 논리적 취약점 및 정합성 이슈

### 1-1. 감사 로그(Audit Log) 유실 취약점 (`ON DELETE CASCADE` 도입 문제)
*   **현상**: `refrigerator_logs` 테이블의 `ingredient_id` 외래 키 제약 조건에 `ON DELETE CASCADE`가 지정되어 있습니다.
    ```sql
    FOREIGN KEY (ingredient_id) REFERENCES ingredients(id) ON DELETE CASCADE
    ```
*   **문제점**: 
    - 14번 쿼리(`DELETE FROM ingredients WHERE status = '폐기대상';`)를 실행해 유통기한이 지난 식재료 데이터를 삭제하면, **해당 식재료와 관련된 모든 반출/소비/폐기 로그가 자동으로 영구 삭제**됩니다.
    - 예를 들어, `양상추`가 썩어서 유저가 '폐기' 처리를 완료하여 로그에 기록되었더라도, 유통기한이 지난 양상추가 `ingredients` 테이블에서 지워지는 순간 **"누가 언제 양상추를 폐기했는지"에 대한 기록이 데이터베이스에서 흔적도 없이 사라집니다.**
    - 이는 데이터베이스 감사(Audit) 및 이력 추적 관점에서 심각한 보안 및 운영 취약점입니다.
*   **개선 제안**:
    1.  **로그 보존 방안 (Snapshot)**: 로그 테이블(`refrigerator_logs`)에 `ingredient_name`과 같은 텍스트 컬럼을 추가해 당시 기록의 스냅샷을 저장하고, `ON DELETE SET NULL` 또는 `ON DELETE NO ACTION`을 고려합니다.
    2.  **소프트 딜리트(Soft Delete)**: `ingredients` 테이블에서 행을 물리적으로 지우지 않고 `is_deleted` 또는 `deleted_at` 컬럼을 활용해 비활성화 처리합니다.

### 1-2. 반출(소비) 처리와 재고 관리의 논리적 불일치
*   **현상**: 샘플 데이터에서 특정 식재료가 반출(`refrigerator_logs.action_type = '반출'`)되었음에도, `ingredients` 테이블에는 상태가 여전히 `'정상'`으로 남아 있습니다.
*   **문제점**:
    - "이미 냉장고 밖으로 꺼내서 먹은 우유(반출 완료)"가 `ingredients` 테이블(현재 냉장고 재고)에 계속 존재하므로, 현재 냉장고에 남은 음식을 조회할 때 오류를 범하게 됩니다.
    - 사용자가 음식을 꺼내 갔다면 재고(`ingredients`)에서 즉시 삭제되거나, `status`가 `'반출됨/소비됨'`으로 업데이트되어야 논리적 정합성이 유지됩니다.
*   **개선 제안**:
    - 음식을 꺼내는 시점에 `ingredients`에서 삭제(`DELETE`)하거나 `status` 컬럼 값을 `'소비'` 또는 `'반출'` 상태로 변경하는 트랜잭션 로직을 정의해야 합니다.

### 1-3. 집계 쿼리의 버그 및 한계 (9번 쿼리)
*   **현상**: 9번 쿼리(사용자별 보관 중인 식재료 총 개수 집계)는 아래와 같이 작성되었습니다.
    ```sql
    SELECT u.name AS 사용자, COUNT(i.id) AS 보관중인_재료수
    FROM users u
    INNER JOIN ingredients i ON u.id = i.user_id
    GROUP BY u.name;
    ```
*   **문제점**:
    1.  **`INNER JOIN` 문제**: 현재 냉장고에 보관 중인 식재료가 0개인 유저(예: 냉장고를 이용하지 않는 입주민)는 결과 화면에서 **아예 누락(조회되지 않음)**됩니다. 셰어하우스 입주민 현황 파악이나 통계 리포트 관점에서 이는 데이터 왜곡입니다.
    2.  **`GROUP BY u.name` 위험성**: 만약 이름이 같은 사용자(동명이인)가 존재할 경우 두 사람의 데이터가 하나로 합쳐져 계산됩니다.
*   **개선 제안**:
    - `LEFT JOIN`을 사용하고, 고유 식별자인 `u.id`로 그룹화하여 동명이인 이슈를 방지하고 0개인 유저도 제대로 표시되도록 개선합니다.
    ```sql
    SELECT u.name AS 사용자, COUNT(i.id) AS 보관중인_재료수
    FROM users u
    LEFT JOIN ingredients i ON u.id = i.user_id
    GROUP BY u.id, u.name;
    ```

## 🛠️ 2. 스키마 설계 및 제약 조건 개선 제안

### 2-1. 값의 유효성 검증을 위한 `CHECK` 제약 조건 누락
*   **문제점**: `ingredients.status`나 `refrigerator_logs.action_type` 등 특정 범주형 데이터를 담는 컬럼에 제약 조건이 없어, 엉뚱한 값(예: `status = '맛있다'`, `action_type = '던지기'`)이 삽입되는 것을 막을 수 없습니다.
*   **개선 제안**: 테이블 생성 시 `CHECK` 제약 조건을 추가하여 애플리케이션 레벨 이외에 DB 레벨에서도 데이터 정합성을 철저히 지킵니다.
    ```sql
    -- ingredients 테이블 예시
    status VARCHAR(20) DEFAULT '정상' CHECK (status IN ('정상', '위험', '폐기대상')),
    
    -- refrigerator_logs 테이블 예시
    action_type VARCHAR(20) NOT NULL CHECK (action_type IN ('추가', '반출', '소비', '폐기')),
    ```

### 2-2. SQLite의 `INT PRIMARY KEY`와 `INTEGER PRIMARY KEY` 차이
*   **설명**: SQLite에서는 `INT PRIMARY KEY`와 `INTEGER PRIMARY KEY`가 다르게 작동합니다.
    - `INTEGER PRIMARY KEY`로 정확히 선언해야만 SQLite 내부의 특수 컬럼인 `rowid`의 앨리어스(Alias)가 되어 **Auto-Increment(자동 증가)** 기능이 기본 적용되며 성능적으로도 최적화됩니다.
    - `INT PRIMARY KEY`로 선언하면 자동 증가 기능이 온전히 작동하지 않고, 사용자가 직접 값을 수동 할당해야 하는 번거로움이 유지됩니다.
*   **개선 제안**: 실서비스 전환 시 편의성과 성능을 위해 `INTEGER PRIMARY KEY AUTOINCREMENT` 또는 최소한 `INTEGER PRIMARY KEY` 스펠링을 적용하는 것이 좋습니다.

### 2-3. 외래 키(Foreign Key) 컬럼의 인덱스 생성 누락
*   **문제점**: 관계형 데이터베이스에서 JOIN이나 CASCADE 삭제 연산이 빈번하게 일어나는 외래 키 컬럼에는 인덱스가 필수적입니다. SQLite는 외래 키 컬럼에 자동으로 인덱스를 생성하지 않으므로, 데이터가 커질수록 `ON DELETE CASCADE` 연산이나 JOIN 연산 시 **전체 테이블 스캔(Full Table Scan)**이 발생해 급격한 성능 저하가 올 수 있습니다.
*   **개선 제안**: 아래와 같이 외래 키 컬럼들에 대해 명시적으로 인덱스를 만들어 줍니다.
    ```sql
    CREATE INDEX idx_ingredients_user_id ON ingredients(user_id);
    CREATE INDEX idx_ingredients_category_id ON ingredients(category_id);
    CREATE INDEX idx_refrigerator_logs_ingredient_id ON refrigerator_logs(ingredient_id);
    CREATE INDEX idx_refrigerator_logs_user_id ON refrigerator_logs(user_id);
    ```

### 2-4. 시간대(Timezone) 정보 및 현재 시각 기본값 관리
*   **설명**: `refrigerator_logs.logged_at`에 `DEFAULT CURRENT_TIMESTAMP`를 사용하고 있습니다. SQLite는 기본적으로 UTC 시간대 기준으로 시각을 기록하므로, 한국 시간(KST, UTC+9)과 9시간의 차이가 발생합니다.
*   **개선 제안**: 한국 기준의 로컬 시간대로 로그를 조회하거나 입력할 수 있도록, 쿼리 레벨에서 `datetime(logged_at, 'localtime')` 형식을 권장하거나 애플리케이션 단에서 명시적으로 현지 시각을 넣어주는 디자인 패턴을 설명해 주면 좋습니다.

## 🌟 3. 최종 요약

이 프로젝트는 입문 단계에서 다루기 힘든 **외래 키 검사 메커니즘(`PRAGMA foreign_keys = ON;`) 작동 방식**과 **서브쿼리 vs JOIN의 성능 및 작동 분석**, **실무 지향적인 위험성 분석 대시보드 지표 쿼리**까지 아주 높은 수준으로 다루고 있습니다.

위에서 지적된 **감사 로그 CASCADE 삭제 취약점**과 **INNER JOIN 집계 누락 오류**를 수정하고 **CHECK 제약 조건**을 적용한다면, 실제 상용화 아키텍처로 바로 전환해도 손색이 없을 만큼 아주 단단하고 완성도 높은 데이터베이스 설계가 될 것입니다. 👍

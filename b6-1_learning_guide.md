## 제 1장. 클라우드와 인프라의 여정: 물리 서버에서 가상화까지

### 1.1 온프레미스(On-Premise) 시절의 서버실 풍경
오늘날 우리는 클릭 몇 번으로 미국, 유럽, 서울에 가상 서버를 띄웁니다. 하지만 불과 10~20년 전만 해도 서비스를 하나 만들기 위해서는 완전히 다른 세상의 고통을 겪어야 했습니다. 이를 **온프레미스(On-Premise)** 또는 자체 구축 인프라 환경이라고 부릅니다.

그 시절, 새로운 서비스를 런칭하라는 지시를 받은 인프라 엔지니어의 일과는 다음과 같았습니다.

```text
[물리 서버 도입의 여정]
기획 및 트래픽 예측 -> 서버 사양 산정(결재) -> 하드웨어 발주(HP, Dell 등) -> 배송 대기(2~4주)
-> IDC(인터넷 데이터 센터) 입고 -> 랙 마운트(무거운 쇳덩이 고정) -> 전원 및 랜 케이블 가닥선 정리 
-> OS(Linux/Windows) CD/USB로 설치 -> 드라이버 세팅 -> 네트워크 대역 설정 -> 서비스 배포
```

이 과정에서 겪는 치명적인 문제점들은 다음과 같았습니다.
1. **시간적 병목 (Time-to-Market)**: 서버 하나를 띄우는 데 결재부터 실제 작동까지 최소 한 달이 걸렸습니다. 시장 변화에 유연하게 대응하는 것은 원천적으로 불가능했습니다.
2. **트래픽 예측의 실패**: 서비스가 대박이 나서 사용자가 폭증하면 서버가 즉시 뻗어버렸고, 새 서버를 주문해서 꼽을 때까지 손가락만 빨아야 했습니다. 반대로 사용자가 전혀 오지 않으면, 수천만 원짜리 장비가 유휴 상태로 방치되어 감가상각만 발생했습니다.
3. **유지 보수의 고통**: 하드웨어는 소모품입니다. 한밤중에 물리 하드디스크가 고장 나거나 파워서플라이가 터지면, 엔지니어는 서버실(IDC)로 방한복(에어컨 바람 때문에 매우 춥습니다)을 입고 랙 사이에 쪼그려 앉아 직접 부품을 갈아 끼워야 했습니다.

### 1.2 가상화(Virtualization) 기술의 등장과 하이퍼바이저
물리 하드웨어의 엄청난 낭비와 비효율을 해결하기 위해 컴퓨터 과학자들은 고민했습니다. 
> "하나의 강력한 물리 컴퓨터 안에서 여러 개의 운영체제(OS)가 마치 독립된 별도의 컴퓨터인 것처럼 속여서 돌릴 수 없을까?"

이 고민에서 탄생한 기술이 바로 **가상화(Virtualization)**입니다. 그리고 물리 하드웨어 위에 올라가 가상 컴퓨터를 창조하고 통제하는 소프트웨어 레이어를 **하이퍼바이저(Hypervisor)**라고 부릅니다.

하이퍼바이저는 하드웨어(CPU, Memory, Disk, Network)를 논리적으로 잘게 쪼개어 가상 머신(Virtual Machine, VM)에 나누어 줍니다. VM 내부의 게스트 OS(Guest OS)는 자신이 진짜 물리 컴퓨터 위에서 독점적으로 돌고 있다고 굳게 믿지만, 실제로는 하이퍼바이저가 분할해 준 일부의 자원만을 쓰고 있는 것입니다.

가상화 덕분에 다음과 같은 마법이 가능해졌습니다.
- **물리 서버의 고효율화**: 하나의 큰 서버를 10개, 20개의 가상 서버로 쪼개어 자원 낭비를 극도로 줄였습니다.
- **가상 서버의 복제와 이동**: 가상 컴퓨터는 결국 호스트 컴퓨터 내부의 '파일(이미지)' 형태에 불과하므로, 복사-붙여넣기를 통해 1초 만에 새로운 서버를 복제해 낼 수 있게 되었습니다.

### 1.3 클라우드 컴퓨팅(Cloud Computing)의 철학: "소유에서 대여로"
AWS(Amazon Web Services)는 이 가상화 기술을 거대하게 확장하여 **"전 세계에 거대한 데이터 센터를 짓고, 하드웨어와 네트워크를 가상화하여 API를 통해 초 단위로 빌려주는 비즈니스"**를 개척했습니다. 이것이 바로 클라우드 컴퓨팅입니다.

클라우드 컴퓨팅의 5대 특징(NIST 정의)은 다음과 같습니다.
1. **주문형 셀프 서비스 (On-demand Self-service)**: 사람의 개입 없이 API 호출이나 웹 클릭 한 번으로 인프라를 즉시 확보합니다.
2. **광범위한 네트워크 접속 (Broad Network Access)**: 어디서나 인터넷만 있으면 접근이 가능합니다.
3. **신속한 탄력성 (Rapid Elasticity)**: 트래픽에 맞춰 실시간으로 서버 대수를 늘리거나 줄일 수 있습니다 (Auto Scaling).
4. **자원의 공유 (Resource Pooling)**: 여러 사용자가 물리 자원을 나누어 쓰지만, 가상화 기술 덕분에 서로 격리되어 안전합니다.
5. **측정 가능한 서비스 (Measured Service)**: 사용한 시간, 사용한 데이터양만큼만 요금을 지불합니다.

이제 여러분은 서버를 "소유(CAPEX - 자본 지출)"하는 시대에서, 사용한 만큼 요금을 지불하는 "대여(OPEX - 운영 지출)"의 시대로 들어왔습니다. 그리고 이번 미션은 그 대여한 가상의 영토 안에 **독자적인 가상 요새(네트워크)**를 짓는 일부터 시작합니다.

## 제 2장. AWS 네트워크의 심장, VPC (Virtual Private Cloud)

### 2.1 Software Defined Networking (SDN)이란 무엇인가?
과거의 네트워크 격리는 매우 까다로웠습니다. 다른 부서나 다른 고객의 트래픽이 섞이지 않게 하려면 물리 라우터와 스위치 장비에서 스위치 포트를 물리적으로 나누거나, VLAN(Virtual LAN) 태그를 설정하고 케이블을 따로 꽂아야 했습니다. 

그러나 AWS 같은 대규모 클라우드 환경에서는 매초 수천 개의 가상 네트워크가 생성되고 삭제됩니다. 이때마다 AWS 엔지니어가 물리 데이터 센터로 달려가 랜 케이블을 꼽을 수는 없는 노릇입니다.

이를 해결하기 위해 등장한 개념이 **SDN (Software Defined Networking)**과 **네트워크 가상화**입니다. AWS는 물리 네트워크 하드웨어 위에 소프트웨어 제어 레이어를 얹었습니다. 패킷의 헤더에 가상 네트워크 식별자를 붙이고 가상 스위치가 이를 소프트웨어적으로 라우팅 및 캡슐화(Encapsulation)하여 배달하는 방식을 사용합니다. 

이러한 고도의 가상 네트워크 기술 위에 구현된, AWS 고객별 독립적인 가상 네트워크 공간이 바로 **VPC (Virtual Private Cloud)**입니다.

VPC는 쉽게 말해 **"AWS라는 거대한 가상의 우주 안에 나만을 위해 만들어진 독점적이고 고립된 영토"**입니다. 내 VPC 안에서 돌아가는 서버들은 다른 고객의 VPC 내 서버들과 철저히 격리되어, 서로 어떠한 트래픽도 주고받을 수 없는 완벽한 보안 환경을 보장받습니다.

### 2.2 IP 주소의 기초: IPv4와 사설 IP vs 공인 IP
네트워크의 기본은 주소입니다. 인터넷 프로토콜 버전 4(IPv4)는 컴퓨터가 서로 통신하기 위해 부여받는 32비트 크기의 고유 주소 체계입니다. 

32비트는 0 또는 1이 32개 있다는 뜻이며, 인간이 읽기 편하도록 8비트씩 4묶음으로 나누어 10진수로 표현합니다. 그것이 바로 우리가 흔히 보는 `192.168.0.1` 같은 형태입니다.
- $2^{32} = 4,294,967,296$ (약 43억 개)

하지만 43억 개의 IP 주소는 전 세계 모든 인류와 기기가 나누어 쓰기엔 턱없이 부족합니다. 이 한계를 극복하기 위해 네트워크 엔지니어들은 주소를 두 종류로 분류했습니다.

1. **공인 IP (Public IP)**: 전 세계 인터넷망에서 유일무이함이 보장되는 주소입니다. 인터넷에 연결된 누구나 이 주소로 찾아올 수 있습니다. 우리가 네이버나 구글에 접속할 때 쓰는 주소들이 바로 공인 IP입니다.
2. **사설 IP (Private IP)**: 폐쇄된 내부 네트워크(사설망) 안에서만 사용되는 주소입니다. 인터넷망에서는 통용되지 않으며, 외부에서 이 주소로 직접 찾아올 수도 없습니다. 

> **[우편물 비유]**
> - **공인 IP**: "대한민국 서울시 강남구 테헤란로 123, OO빌딩" (전 세계 어디서든 편지를 보낼 수 있는 고유한 주소)
> - **사설 IP**: "3층 302호 김 대리의 책상" (회사 내부에서는 다 알아듣지만, 미국에서 편지 봉투에 '김 대리 책상'이라고 적어 보내면 절대 배달되지 않는 주소)

VPC는 나만의 사설망을 구축하는 것이므로, 내부에서 사용할 **사설 IP 대역**을 직접 정의해야 합니다. 이 사설 IP 대역은 국제 표준(RFC 1918)에 의해 인터넷망과 절대 겹치지 않도록 미리 예약되어 있습니다.

* **Class A**: `10.0.0.0` ~ `10.255.255.255` (가장 많은 주소 보유 가능)
* **Class B**: `172.16.0.0` ~ `172.31.255.255`
* **Class C**: `192.168.0.0` ~ `192.168.255.255` (가정용 공유기에서 흔히 쓰는 대역)

AWS에서 VPC를 만들 때는 보통 `10.0.0.0` 대역을 널리 사용합니다.

### 2.3 CIDR(Classless Inter-Domain Routing)과 서브넷 마스크 연산의 원리
VPC를 설정할 때 가장 먼저 맞닥뜨리는 암호 같은 표기법이 있습니다. 바로 **CIDR (Classless Inter-Domain Routing, 사이더)** 표기법입니다. 예컨대 `10.0.0.0/16` 같은 형태입니다. 이 표기의 수학적 원리를 완벽하게 분해해 봅시다.

`10.0.0.0/16`에서 슬래시(`/`) 뒤에 붙은 숫자 `16`은 **"32비트 IP 주소 중 왼쪽에서부터 16비트까지를 '네트워크 식별자(Network ID)'로 고정하겠다"**는 선언입니다. 이를 **서브넷 마스크(Subnet Mask)**라고 부릅니다. 나머지 비트인 $32 - 16 = 16$비트는 자유롭게 컴퓨터(호스트)들에 나누어 줄 수 있는 주소 범위가 됩니다.

이것을 이진수로 풀어서 시각화해 보겠습니다.

```text
IP 주소: 10.0.0.0 -> 이진수 변환:
10       . 0        . 0        . 0
00001010 . 00000000 . 00000000 . 00000000

서브넷 마스크 (/16 -> 왼쪽부터 16개의 비트를 1로 채움):
11111111 . 11111111 . 00000000 . 00000000 (이진수)
255      . 255      . 0        . 0        (10진수 표현)
```

이 고정된 16비트 구역 뒤의 16비트는 `00000000.00000000`($0$)부터 `11111111.11111111`($65535$)까지 가변적입니다.
따라서 `10.0.0.0/16`이 표현하는 전체 IP 주소의 개수는 다음과 같습니다.
- $2^{16} = 65,536$개
- 범위: `10.0.0.0` ~ `10.0.255.255`

만약 `10.0.0.0/24`를 쓴다면 어떨까요?
왼쪽에서부터 24비트를 고정하겠다는 의미이므로, 마스크는 `255.255.255.0`이 됩니다.
자유롭게 쓸 수 있는 공간은 마지막 8비트뿐이므로, 할당 가능한 IP 개수는 다음과 같습니다.
- $2^{8} = 256$개
- 범위: `10.0.0.0` ~ `10.0.0.255`

> **[슬래시(/) 뒤의 숫자가 커질수록 네트워크 크기는 작아집니다!]**
> - `/16` : 65,536개의 IP 대규모 영토
> - `/24` : 256개의 IP 소규모 구역
> - `/32` : 단 1개의 특정 IP 주소 ($2^0=1$)

이번 미션에서는 보통 VPC 전체 대역으로 넓은 영토인 `10.0.0.0/16`을 할당하고, 이 안에서 필요한 만큼의 크기로 쪼개어 서브넷들을 만들어 쓸 것입니다.

## 제 3장. 네트워크의 세포 분열, Subnet (서브넷)

### 3.1 서브넷의 본질적 존재 이유
VPC라는 거대한 영토(`10.0.0.0/16`, 65,536개의 IP)를 통째로 하나의 거대한 방으로 쓰면 어떤 일이 벌어질까요?

네트워크에 연결된 기기들은 종종 "네트워크 전체에 소리를 지르는" **브로드캐스트(Broadcast)** 통신을 합니다. 예컨대 "IP 주소 `10.0.0.5`인 녀석 누구야? 손들어봐!"라고 외치는 ARP(Address Resolution Protocol) 패킷 같은 것입니다. 

만약 6만 대의 서버가 한 방에 모여 있다면, 한 대가 소리칠 때마다 나머지 59,999대의 서버가 그 불필요한 외침을 수신하고 처리하느라 CPU 성능과 네트워크 대역폭이 낭비될 것입니다. 이를 **브로드캐스트 도메인(Broadcast Domain)**의 비대화 문제라고 합니다.

또한 보안 관점에서도 끔찍합니다. 내부 결제 시스템을 다루는 극비 데이터베이스 서버와 일반 사용자가 접속하는 퍼블릭 웹 서버가 같은 방에 무방비하게 섞여 있다면, 웹 서버가 해킹당하는 순간 데이터베이스 서버 역시 완전히 무방비하게 털리게 됩니다.

이 때문에 우리는 거대한 VPC 영토 안에 **"보안 벽과 파티션(라우팅 경계)"**을 쳐서 네트워크 영역을 쪼개게 되는데, 이 쪼개진 방들을 **서브넷 (Subnet, 하위 네트워크)**이라고 부릅니다.

```text
[VPC 영토 10.0.0.0/16]
┌──────────────────────────────────────────────────────────────┐
│  ┌───────────────────────────┐    ┌───────────────────────┐  │
│  │ Subnet A (10.0.1.0/24)    │    │ Subnet B (10.0.2.0/24)│  │
│  │ 웹 서버 배치 (퍼블릭)      │    │ DB 서버 배치 (프라이빗)│  │
│  └───────────────────────────┘    └───────────────────────┘  │
└──────────────────────────────────────────────────────────────┘
```

### 3.2 Public Subnet vs Private Subnet: 물리적 실체의 진실
AWS 콘솔에서 서브넷을 생성할 때, '퍼블릭 서브넷' 버튼과 '프라이빗 서브넷' 버튼이 따로 있는 것이 아닙니다. 
인프라를 처음 공부하시는 분들은 종종 AWS 어딘가에 물리적으로 '퍼블릭 전용 기계'와 '프라이빗 전용 기계'가 따로 존재한다고 오해하곤 합니다.

하지만 실상은 완전히 동일한 가상 스위치와 하드웨어 레이어에서 돌아갑니다. 두 서브넷을 구분 짓는 유일한 차이는 **"해당 서브넷과 연결된 라우팅 테이블(Route Table)이 외부 인터넷과 연결되는 통로(Internet Gateway)로 향하는 길을 알고 있는가 없는가"** 하는 논리적 설정의 차이입니다.

* **퍼블릭 서브넷 (Public Subnet)**: 외부 인터넷 세상과 양방향 통신이 가능한 인터넷 게이트웨이(IGW)로 향하는 이정표(라우팅 규칙)가 설정된 서브넷. 외부의 사용자가 이 서브넷 안의 서버(예: EC2)에 직접 접속할 수 있습니다.
* **프라이빗 서브넷 (Private Subnet)**: 외부 인터넷 게이트웨이로 가는 라우팅 경로가 없는 서브넷. 오직 VPC 내부의 다른 서브넷들과만 통신할 수 있으며, 외부 인터넷 세상에서는 절대로 이 서브넷 안에 있는 서버에 직접 접근할 수 없습니다.

이번 미션에서는 외부 인터넷 사용자가 우리가 만든 웹 서버에 들어와야 하므로, **Public Subnet**을 하나 만들고 그 안에 웹 서버 역할을 할 EC2 인스턴스를 올려놓게 됩니다.

### 3.3 AWS 서브넷에서 사라진 5개의 예약 IP 주소
서브넷 대역을 `10.0.1.0/24`로 설정하면 총 256개의 주소를 쓸 수 있을 것 같지만, 실제로 가용 가능한 주소는 **251개**뿐입니다. AWS는 각 서브넷을 생성할 때마다 맨 앞의 4개 주소와 맨 뒤의 1개 주소를 내부 시스템 관리를 위해 강제로 예약해 둡니다. 이는 매우 중요한 네트워크 기밀입니다.

`10.0.1.0/24` 서브넷을 기준으로 예약된 주소와 그 역할은 다음과 같습니다.

1. **`10.0.1.0` (네트워크 주소)**: 해당 서브넷 자체를 대표하는 이름표로 사용되는 주소입니다. 호스트에 할당할 수 없습니다. (전통적인 네트워크 표준)
2. **`10.0.1.1` (VPC 라우터 주소)**: 서브넷 내부의 기기들이 다른 서브넷이나 바깥세상과 통신할 때 거쳐 가는 기본 게이트웨이(Default Gateway)의 가상 IP 주소입니다.
3. **`10.0.1.2` (AWS DNS 서버 주소)**: 도메인 이름을 IP 주소로 바꾸어주는 AWS 제공 DNS(AmazonProvidedDNS)의 IP 주소입니다. (항상 서브넷 대역의 2번째 주소로 예약됩니다)
4. **`10.0.1.3` (미래 대비 예약 주소)**: AWS가 향후 새로운 내부 서비스를 제공할 때 쓰기 위해 선점해 둔 예비 주소입니다.
5. **`10.0.1.255` (네트워크 브로드캐스트 주소)**: 서브넷 내 모든 기기에 동시에 데이터를 쏘는 브로드캐스트를 위해 전통적으로 사용되는 주소입니다. 비록 VPC 안에서는 브로드캐스트가 지원되지 않지만, 표준 준수를 위해 비워 둡니다.

따라서 여러분이 인프라를 설계할 때 서버 수량을 극도로 빡빡하게 계산한다면 이 '사라진 5개의 주소' 때문에 IP가 모자라는 대참사가 날 수 있으므로 주의해야 합니다.

## 제 4장. 패킷의 나침반, Route Table (라우팅 테이블)

### 4.1 라우터와 이정표의 원리
우리가 사는 도로망에는 교차로마다 표지판이 서 있습니다. "강남 방면은 직진, 잠실 방면은 우회전." 
네트워크 패킷의 세계에서도 마찬가지입니다. 내 서버에서 출발한 데이터 조각(패킷)이 목적지 IP를 들고 가상 네트워크 카드를 떠날 때, 이 패킷을 받아 "너는 이쪽 길로 가야 해"라고 방향을 지시해 주는 길잡이 장치가 **라우터(Router)**이며, 라우터가 참조하는 이정표 지도가 바로 **라우팅 테이블(Route Table)**입니다.

라우팅 테이블은 단순한 매핑 표입니다.
> **"목적지 IP 주소가 [A 대역]에 속한다면, [B 대상]으로 패킷을 던져라."**

### 4.2 로컬 라우팅(Local Route)의 기본 규칙
VPC 내부에서 서브넷을 만들면, 그 서브넷의 라우팅 테이블에는 아무 짓도 하지 않아도 무조건 가장 먼저 등록되어 있고 절대 삭제할 수 없는 절대적인 규칙이 존재합니다. 그것이 바로 **로컬 라우팅(Local Route)** 규칙입니다.

```text
[VPC의 기본 라우팅 테이블 예시]
Destination (목적지)       Target (대상)
10.0.0.0/16                local
```

이 규칙의 의미는 다음과 같습니다.
> "어떤 패킷이든 목적지 IP가 `10.0.0.0/16`(즉, 우리 VPC의 영토 안)에 해당하는 주소라면, 외부로 나가지 말고 **VPC 내부(local)의 가상 스위치 네트워크**를 타고 직접 배달해라."

이 규칙 덕분에 VPC 안의 여러 서브넷들은 서로 다른 방에 쪼개져 있더라도 라우팅 테이블을 통해 안전하고 신속하게 사설 IP 통신을 주고받을 수 있습니다.

### 4.3 가장 구체적인 경로 규칙: Longest Prefix Match (LPM)
만약 라우팅 테이블에 다음과 같이 겹치거나 충돌하는 규칙이 여러 개 들어있다면 라우터는 어떤 길을 선택할까요?

```text
목적지 패킷의 목적지 IP: 10.0.1.55

[라우팅 테이블 규칙들]
규칙 1: 10.0.0.0/16  -> 대상: local
규칙 2: 10.0.1.0/24  -> 대상: 어떤 특정 인스턴스 (ENI)
규칙 3: 0.0.0.0/0    -> 대상: 인터넷 게이트웨이 (igw-xxxx)
```

이 주소 `10.0.1.55`는 세 규칙의 범위에 모두 맞아떨어집니다. 
- `0.0.0.0/0`은 모든 주소를 의미하므로 맞습니다.
- `10.0.0.0/16`도 `10.0.x.x`를 커버하므로 맞습니다.
- `10.0.1.0/24` 역시 `10.0.1.x`를 커버하므로 맞습니다.

네트워크 라우터는 이때 **LPM (Longest Prefix Match, 가장 긴 프리픽스 매칭)** 원칙을 따릅니다. 즉, **"서브넷 마스크 비트 수(슬래시 뒤의 숫자)가 가장 큰, 즉 가장 좁고 구체적으로 대상을 묘사한 경로 규칙을 최우선으로 적용한다"**는 규칙입니다.

위의 예시에서는 `/24`가 `/16`이나 `/0`보다 훨씬 구체적이므로, 라우터는 고민 없이 **규칙 2**를 선택해 패킷을 배달합니다. 

### 4.4 모든 곳을 향하는 통로: `0.0.0.0/0` (기본 게이트웨이)
라우팅 테이블에서 가장 중요한 특수 주소가 바로 `0.0.0.0/0`입니다. 
서브넷 마스크 비트 수가 `0`이라는 뜻은 **"아무런 비트도 고정하지 않겠다"**는 의미로, **이 세상에 존재하는 모든 IPv4 주소**를 지칭하는 와일드카드 주소입니다. 이를 전문 용어로 **기본 경로(Default Route)** 또는 **기본 게이트웨이**라고 부릅니다.

우리가 퍼블릭 서브넷을 만들고 내부 인스턴스가 바깥 인터넷 세상(네이버, 구글 등)에 있는 서버들과 통신하게 하려면, 라우팅 테이블에 반드시 아래와 같은 규칙을 손수 적어주어야 합니다.

```text
Destination (목적지)       Target (대상)
10.0.0.0/16                local          (내부 통신용)
0.0.0.0/0                  igw-xxxxxxxxx  (인터넷으로 나가는 통로)
```

이 라우팅 테이블을 해석해 봅시다.
1. 만약 패킷이 `10.0.5.2`로 가려고 한다면? `/16` 범위에 맞으므로 `local`로 향합니다. (VPC 내부 통신)
2. 만약 패킷이 네이버 서버인 `223.130.195.200`으로 가려고 한다면? `10.0.0.0/16` 범위에 맞지 않으므로 다음 규칙을 봅니다. `0.0.0.0/0` 범위에는 우주의 모든 IP가 들어가므로 여기에 매칭되어 **인터넷 게이트웨이(igw)**를 타고 바깥 인터넷으로 탈출하게 됩니다.

이 단순한 줄 하나가 서브넷을 **퍼블릭 서브넷**으로 만드는 핵심 스위치입니다. 만약 이 줄이 빠져 있다면, 그 서브넷은 외부 인터넷망과 단절된 **프라이빗 서브넷**이 됩니다.

## 제 5장. 세상과 소통하는 관문, Internet Gateway (IGW)

### 5.1 사설 영토와 문명 세계를 잇는 다리
VPC는 완벽히 고립된 영토라고 했습니다. 사설 IP 대역(`10.0.0.0/16`)만을 사용하기 때문에, 바깥 인터넷 세상의 라우터들은 이 주소를 전혀 이해하지 못하며 이 주소로 찾아올 수도 없습니다.

이 폐쇄된 영토에 인터넷이라는 광활한 바다로 나갈 수 있는 항구를 만들어주는 리소스가 바로 **인터넷 게이트웨이 (Internet Gateway, IGW)**입니다.

인터넷 게이트웨이는 다음과 같은 핵심 기능을 담당합니다.
1. **외부 통로 제공**: VPC 내부의 자원들이 바깥 인터넷에 접속하고, 바깥 인터넷 사용자들도 VPC 내부로 유입될 수 있는 관문이 됩니다.
2. **1:1 Static NAT (Network Address Translation) 처리**: 이것이 바로 IGW의 핵심 마법입니다.

### 5.2 1:1 Static NAT의 내부 원리
여기서 중요한 의문이 생깁니다.
> "내 EC2 인스턴스의 진짜 주소는 사설 IP인 `10.0.1.10`인데, 어떻게 바깥 세상의 사용자가 내 서버에 찾아올 수 있고, 내 서버도 네이버에 접속해 데이터를 긁어올 수 있는 걸까?"

이것을 가능하게 하는 기술이 바로 **NAT(네트워크 주소 변환)**입니다. 

여러분이 EC2 인스턴스를 생성할 때 "퍼블릭 IP 자동 할당"을 켜면, AWS는 글로벌 공인 IP 풀(Pool)에서 남는 공인 IP 주소 하나(예: `54.180.12.34`)를 임시로 건져내어 여러분의 EC2에 부여합니다. 

흥미롭게도, 실제로 EC2 서버 내부로 들어가 `ifconfig`나 `ip addr` 명령으로 네트워크 카드를 조회해 보면 **자신의 공인 IP인 `54.180.12.34`는 눈을 씻고 찾아봐도 보이지 않습니다.** 오직 사설 IP인 `10.0.1.10`만이 설정되어 있을 뿐입니다.

공인 IP와 사설 IP 사이의 매핑 정보는 오직 **인터넷 게이트웨이(IGW)**만이 머릿속에 기억하고 관리합니다.

```text
[외부 사용자]                                      [인터넷 게이트웨이 (IGW)]                             [EC2 인스턴스]
공인 IP 사용                                      (주소 변환기)                                         사설 IP 사용
  │                                                    │                                                    │
  │─── [패킷 송신] ───────────────────────────────────>│─── [사설 IP로 변환] ───────────────────────────────>│
  │    출발지: 211.200.5.5 (사용자)                    │    출발지: 211.200.5.5                              │
  │    목적지: 54.180.12.34 (EC2 공인)                 │    목적지: 10.0.1.10 (EC2 사설)                      │
  │                                                    │                                                    │
  │<── [패킷 수신] ────────────────────────────────────│<── [공인 IP로 변환] ───────────────────────────────│
  │    출발지: 54.180.12.34 (EC2 공인)                 │    출발지: 10.0.1.10 (EC2 사설)                     │
  │    목적지: 211.200.5.5                             │    목적지: 211.200.5.5                              │
```

1. **외부에서 들어올 때(Inbound)**: 외부 사용자가 `54.180.12.34`를 향해 패킷을 보냅니다. 패킷이 AWS 네트워크 경계에 있는 IGW에 도달하면, IGW는 "아하! `54.180.12.34`는 내 장부에 적힌 `10.0.1.10` 사설 IP를 뜻하는군!" 하고 패킷의 목적지 주소를 `10.0.1.10`으로 바꿔서 VPC 내부로 통과시킵니다.
2. **내부에서 나갈 때(Outbound)**: EC2가 외부 사이트로 패킷을 내보냅니다. 출발지 주소에 자신의 사설 IP인 `10.0.1.10`을 적어서 보냅니다. 이 패킷이 라우팅 테이블을 타고 IGW를 통과할 때, IGW는 "바깥세상 라우터들은 이 사설 IP를 모르니까 내가 가진 공인 IP인 `54.180.12.34`로 출발지 주소를 가꿔서 내보내 줄게!" 하며 패킷을 변환해 인터넷 바다로 던져줍니다.

이것을 **1:1 양방향 정적 주소 변환(Static NAT)**이라고 부릅니다. 이 과정 덕분에 사설망 내부의 서버가 인터넷과 안전하게 섞여서 통신할 수 있는 것입니다.

### 5.3 IGW의 아키텍처적 비밀: 단일 장애점(SPOF)이 아닌 이유
우리는 VPC를 만들고 인터넷 게이트웨이를 단 1개만 생성해서 연결합니다. 
인프라 설계를 공부하다 보면 이런 의문이 듭니다. 
> "모든 트래픽이 저 물리적 관문인 IGW 하나로만 쏠린다면, 병목 현상이 생기거나 저 장비가 폭발하면 서비스가 전체 마비되는 단일 장애점(Single Point of Failure, SPOF)이 되지 않나요?"

정답은 **"아닙니다"** 입니다. 

AWS의 인터넷 게이트웨이는 이름만 '게이트웨이'일 뿐, 창고에 서 있는 단일 물리 장비나 가상 머신이 아닙니다. 
사실 IGW는 **AWS 소프트웨어 정의 네트워크(SDN) 상에 널리 분산되어 논리적으로 동작하는 고가용성, 초확장성 가상 커넥터**입니다.

실제 트래픽은 데이터 센터 내의 수많은 물리 스위치 노드들을 통해 고도로 분산되어 처리되므로, 대역폭의 한계가 없고 하드웨어 결함으로 인해 IGW가 죽는 일 또한 구조적으로 일어나지 않습니다. AWS가 우리에게 제공하는 완벽한 가상화 추상화의 이점입니다.

## 제 6장. 가상 영토의 일꾼, EC2 (Elastic Compute Cloud)

### 6.1 가상 서버(Compute)의 본질과 하이퍼바이저 가상화
네트워크망(VPC, Subnet, Route Table, IGW)을 완벽하게 깔았다면, 이제 그 도로망 위에 실제 웹 애플리케이션 코드를 실행하고 손님을 맞이할 진짜 가상 컴퓨터를 얹어야 합니다. 그것이 바로 **EC2 (Elastic Compute Cloud)**입니다.

EC2는 AWS가 물리 데이터 센터(IDC)에 가득 채워둔 거대한 물리 서버 랙 장비들(CPU 코어가 수백 개씩 꽂히고 메모리가 수십 TB 수준인 괴물 장비들)의 하드웨어 자원을 **하이퍼바이저**를 통해 가상으로 슬라이스하여 나누어주는 논리 컴퓨터입니다.

* **Elastic(탄력적인)**: 몇 초 만에 CPU와 RAM 용량을 자유자재로 늘리거나 줄일 수 있고, 필요 없으면 즉시 폐기할 수 있기에 '탄력적'이라고 불립니다.

현재 AWS는 자체 개발한 맞춤형 ASIC 기반 하드웨어 가상화 엔진인 **AWS Nitro(나이트로) 시스템**을 사용하여 가상화 오버헤드를 극도로 줄임으로써, 가상 서버임에도 실제 물리 서버와 거의 100%에 가까운 순수 하드웨어 성능을 끌어내고 있습니다.

### 6.2 인스턴스 이름의 비밀 분석 (예: `t3.micro`)
EC2 인스턴스를 고를 때 우리는 `t2.micro`, `t3.micro`, `c5.large` 같은 이름을 마주합니다. 이 외계어 같은 코드명 속에는 하드웨어의 특성과 등급이 정밀하게 설계되어 있습니다.

```text
  t      3    .   micro
 ───    ───      ───────
  │      │          │
  │      │          └─ 인스턴스 크기 (Size - CPU, RAM의 양)
  │      └─ 인스턴스 세대 (Generation - 숫자가 높을수록 신형 칩셋과 아키텍처)
  └─ 인스턴스 패밀리 (Family - CPU 집중형, 메모리 집중형 등 컴퓨터의 성격)
```

1. **인스턴스 패밀리 (Family)**:
   - **T 패밀리**: 범용(General Purpose) 버스터블 성능 인스턴스. 평소에는 CPU 사용량이 낮지만, 가끔씩 필요할 때 폭발적으로 성능을 끌어올릴 수 있는 알뜰형 인스턴스입니다. (실습용 프리티어 단골 타깃)
   - **C 패밀리**: 컴퓨팅 최적화(Compute Optimized). 고성능 연산, 과학 계산, 머신러닝 학습 등에 적합합니다.
   - **R 패밀리**: 메모리 최적화(Memory Optimized). 대규모 데이터베이스나 캐시(Redis) 서비스에 쓰입니다.

2. **인스턴스 세대 (Generation)**:
   - `t2`보다 `t3`가 인텔/AMD/Graviton의 최신 세대 CPU 칩셋을 사용하여 가격 대비 깡성능과 네트워크 대역폭이 훨씬 우수합니다.

3. **크기 (Size)**:
   - `micro` < `small` < `medium` < `large` < `xlarge` < `2xlarge` 순으로 바지를 고르듯 체급이 올라갑니다.
   - 체급이 한 단계 올라갈 때마다 CPU 코어 수와 Memory 용량, 그리고 가격도 정확히 **2배씩** 정비례하여 증가합니다.

### 6.3 T 패밀리의 핵심 메커니즘: CPU 크레딧(Credit)과 Burstable의 함정
우리가 실습에 사용하는 `t2.micro` / `t3.micro` 인스턴스는 매우 저렴하며 프리티어 범위 안에서 무료로 쓸 수 있습니다. 하지만 이 T 패밀리에는 **"CPU 크레딧"**이라는 독특한 은행 대출 같은 시스템이 숨겨져 있어, 현업에서 이 원리를 모르고 배포했다가 서비스가 마비되는 참사가 빈번하게 일어납니다.

T 패밀리는 **Burstable(버스터블 - 일시적으로 튀어오를 수 있는)** 인스턴스입니다. 
* 가령 `t3.micro`는 평상시 CPU를 **평균 5~10% 수준**만 쓸 것이라 가정하고 저렴하게 파는 상품입니다.
* 서버가 놀고 있을 때(CPU 사용률이 기준치 이하일 때), 계정에는 매분 **'CPU 크레딧'**이라는 포인트가 차곡차곡 적립됩니다.
* 갑자기 사용자가 몰려 CPU 사용률을 80%로 끌어올려야 하면, 서버는 적립해 둔 'CPU 크레딧' 포인트를 초당 일정량씩 소모해가며 폭발적인 100% 성능을 발휘합니다.

**[함정]**: 
만약 서비스 트래픽이 계속 높아서 모아둔 CPU 크레딧을 전부 탕진해 버리면(0이 되면), 인스턴스의 CPU 성능은 하드웨어적으로 **기본 베이스라인(Base Performance, 약 5% 수준)으로 강제 스로틀링(제한)**당합니다. 
이 상태가 되면 서버는 숨을 헐떡이며 극도로 느려지고, 웹 응답 시간이 초과되어 서비스 불능 상태에 빠지게 됩니다.

따라서 실습 중 대규모 스트레스 테스트를 하거나 실제 상용 서비스를 운영할 때는 T 패밀리의 크레딧 추이를 항상 관찰해야 하며, 지속적인 고성능이 요구되는 경우엔 크레딧 개념이 없는 범용 M 패밀리나 C 패밀리 서버로 이주해야 합니다.

### 6.4 EBS (Elastic Block Store): 가상 컴퓨터의 하드디스크
컴퓨터에 본체(EC2)가 있다면 데이터를 영구히 저장할 하드디스크나 SSD가 필요합니다. AWS에서는 이 하드디스크를 **EBS (Elastic Block Store)**라고 부릅니다.

EBS의 매우 독특한 물리적 사실은 다음과 같습니다.
> **"EBS는 EC2 메인보드에 꽂혀 있는 물리 드라이브가 아니라, 네트워크로 멀리 떨어져서 연결된 가상 스토리지 디바이스이다."**

EBS는 고속 네트워크(SAN)를 통해 가상 컴퓨터(EC2)와 연결됩니다. 이 덕분에 다음과 같은 무시무시한 기능이 가능해집니다.
- **분리 및 이동**: EC2 인스턴스가 고장 나서 부서져도, 하드디스크에 해당하는 EBS만 쏙 빼내어(Detach) 옆에 있는 건강한 다른 EC2에 즉시 꽂아(Attach) 데이터를 완벽히 보존할 수 있습니다.
- **스냅샷(Snapshot)**: 하드디스크의 현재 상태 그대로 사진을 찍어(복제본을 생성) 언제든 백업하거나 다른 리전에 똑같은 드라이브를 찍어낼 수 있습니다.

### 6.5 SSH(Secure Shell)와 비대칭키(Key Pair) 암호화 동작 원리
리눅스 가상 서버(EC2)를 생성하면 우리는 모니터도 키보드도 없는 지구 반대편의 검은 화면 컴퓨터에 원격 접속해야 합니다. 이때 사용하는 표준 프로토콜이 **SSH (Secure Shell, 보안 셸 - 22번 포트)**입니다.

AWS는 해커들이 서버를 무차별 공격하는 것을 막기 위해 비밀번호 로그인 방식을 막아두고, **비대칭키 암호화(Asymmetric Key Cryptography)** 기반의 키페어(Key Pair) 인증을 필수적으로 요구합니다.

이 암호학적 동작 방식을 이해해 봅시다. 키페어는 한 쌍의 짝꿍으로 이루어져 있습니다.

```text
[비대칭키(RSA)의 환상의 짝꿍 구조]
1. 프라이빗 키 (Private Key, .pem 파일) -> 나만 가짐 (유출 금지!)
2. 퍼블릭 키 (Public Key)            -> 세상에 공개 (EC2 서버 내부에 미리 저장)
```

* **비대칭키의 원리**: 퍼블릭 키(Public Key)로 자물쇠를 잠근 데이터는 오직 짝꿍인 프라이빗 키(Private Key)로만 열 수 있습니다. 그 반대도 마찬가지입니다.
* **접속 과정**:
  1. EC2 인스턴스가 생성될 때, AWS는 우리가 내려받은 키페어의 **퍼블릭 키** 정보를 EC2 내부의 특정 폴더 파일 (`~/.ssh/authorized_keys`) 안에 고이 박아놓습니다.
  2. 우리가 내 터미널에서 `ssh -i my-key.pem ubuntu@54.180.12.34` 명령을 날립니다.
  3. EC2 서버는 "너 진짜 내 짝꿍 맞니?" 하며 무작위 난수 챌린지 텍스트를 만들어 자신의 **퍼블릭 키**로 잠근 뒤 우리 컴퓨터로 던집니다.
  4. 우리 컴퓨터의 SSH 클라이언트는 내가 가진 **프라이빗 키(`.pem`)**로 그 자물쇠를 풀어서 원래 난수 값을 얻어낸 뒤, 이를 입증하는 서명값을 돌려줍니다.
  5. EC2 서버가 서명을 확인하고 "아! 너 진짜 내 유일한 짝꿍 `.pem`을 가진 주인이 맞구나!" 하며 문을 열어 쉘(Shell) 접속을 허용합니다.

그렇기 때문에 `.pem` 파일(프라이빗 키)은 절대로 잃어버려서도 안 되고(잃어버리면 복구가 원천적으로 불가능해 서버에 영영 들어가지 못하게 됩니다), GitHub이나 외부 커뮤니티에 유출해서도 안 되는 생명줄과 같은 파일입니다.

## 제 7장. 가상 네트워크의 경계 초소, Security Group (보안 그룹)

### 7.1 인스턴스의 철통 방화벽, 보안 그룹
네트워크도 있고 서버도 띄웠습니다. 그런데 만약 전 세계의 온갖 크래커와 해커, 자동화 스팸 봇들이 내 서버의 22번 SSH 포트나 웹 서버 포트로 무차별 공격을 퍼부으면 서버는 즉시 무너집니다.

이를 막기 위해 AWS가 제공하는 가장 즉각적인 가상 방화벽 시스템이 **보안 그룹 (Security Group, SG)**입니다.

보안 그룹은 **"EC2 인스턴스의 가상 네트워크 카드(ENI)를 감싸고 있는 가상의 투명 방패"**라고 이해하면 됩니다. 트래픽이 EC2 컴퓨터 내부 OS로 침투하기 직전, 하이퍼바이저 레이어에서 사전에 유해 패킷을 걸러내 줍니다.

```text
외부 사용자 ──> ( [보안 그룹 방패] ──> [EC2 인스턴스 (OS)] )
                   - 허용 규칙 검사
                   - 차단 규칙 없음 (기본 차단)
```

### 7.2 보안 그룹의 핵심 규칙: "화이트리스트(Whitelist)"
보안 그룹은 철저하게 **화이트리스트(Whitelist)** 방식으로 동작합니다. 
* **화이트리스트**: "기본적으로 모든 문은 꽁꽁 잠겨 있다(Default Deny). 오직 내가 명시적으로 적어놓은 신뢰할 수 있는 규칙에 부합하는 트래픽만 통과시켜 준다."
* 보안 그룹 설정 창에는 '허용(Allow) 규칙'만 존재하며, '차단(Deny) 규칙'을 임의로 등록하는 칸은 존재하지 않습니다. 규칙에 없는 모든 트래픽은 가차 없이 차단되어 드롭(Drop)됩니다.

### 7.3 인바운드(Inbound) vs 아웃바운드(Outbound) 트래픽
* **인바운드 규칙 (Inbound Rules)**: 밖에서 서버로 **들어오는** 통신을 제어합니다.
  - 예: 일반 인터넷 사용자가 내 웹 서버로 들어오게 허용 (`HTTP 80` 포트 오픈)
  - 예: 내가 관리자 목적으로 서버에 접속하게 허용 (`SSH 22` 포트 오픈)
* **아웃바운드 규칙 (Outbound Rules)**: 서버 안에서 바깥세상으로 **나가는** 통신을 제어합니다.
  - 예: 내 서버가 업데이트 패키지를 다운받기 위해 외부의 우분투 저장소 사이트에 요청을 보냄
  - 보안 그룹을 생성하면 기본적으로 아웃바운드는 모든 대상(`0.0.0.0/0`)에 대해 전체 허용되어 있습니다.

### 7.4 Stateful(상태 유지)의 놀라운 기술적 원리
보안 그룹을 이해할 때 가장 중요한 컴퓨터 과학 단어가 바로 **Stateful (상태 유지)**입니다. 
이 개념을 완벽하게 이해하기 위해, 반대 개념인 **Stateless (상태 비저장)**를 함께 비교해야 합니다.

* **Stateless (상태 비저장 - 예: 네트워크 ACL)**: 
  나가는 길과 들어오는 길을 완전히 별개로 취급합니다. 예컨대 "80포트 인바운드를 허용했으니 들어올 수 있다" 하더라도, 들어왔던 패킷이 일을 마치고 되돌아 나갈 때 "80포트 아웃바운드가 차단되어 있다면" 응답을 돌려주지 못해 통신이 깨집니다. 들어오는 문과 나가는 문에 각각 통행증을 끊어주어야 합니다.
  
* **Stateful (상태 유지 - 보안 그룹)**:
  보안 그룹은 패킷이 한번 들어오거나 나갈 때, 하이퍼바이저 수준에서 그 통신의 **'연결 상태(Connection)'를 기억하는 '커넥션 추적 테이블(Connection Tracking Table)'**에 기록해 둡니다.

```text
[보안 그룹 커넥션 추적 테이블 예시]
출발지 IP:포트          목적지 IP:포트          상태(State)
211.200.5.5:54321      10.0.1.10:80           ESTABLISHED
```

만약 외부 사용자(`211.200.5.5`)가 `HTTP 80` 포트로 들어오는 것이 인바운드 규칙에 의해 허용되어 들어왔다면, 보안 그룹은 이 연결을 메모리에 등록해 둡니다. 

이후 서버가 응답 패킷을 만들어 사용자에게 다시 되돌려 보낼 때, 보안 그룹은 **아웃바운드 규칙을 전혀 보지 않습니다!** 커넥션 추적 테이블을 쓱 보고 "아하, 방금 합법적으로 들어와서 대화 중인 녀석이구나!" 하고 그냥 프리패스로 통과시켜 줍니다.

반대로, 내 서버가 외부 구글 서버(`8.8.8.8`)에 `curl` 요청을 보내기 위해 아웃바운드로 나갔다면, 되돌아 들어오는 구글의 응답 패킷 역시 **인바운드 규칙(비록 80포트 등이 막혀있더라도)을 보지 않고** 무조건 들여보내 줍니다.

이 Stateful 원리 덕분에 우리는 복잡하게 나가는 통신에 대한 방화벽 규칙을 일일이 매칭하지 않고도 아주 쉽고 유연하게 방화벽 정책을 세울 수 있습니다.

### 7.5 보안 그룹 vs 네트워크 ACL(NACL) 결정적 차이
면접 단골 질문이자 아키텍처 설계 시 가장 헷갈리는 두 요소의 비교표를 머리에 박아 넣읍시다.

| 비교 항목 | 보안 그룹 (Security Group) | 네트워크 ACL (Network ACL) |
| :--- | :--- | :--- |
| **작동 대상** | **인스턴스 레벨** (네트워크 카드 ENI) | **서브넷 레벨** (서브넷 경계선) |
| **상태 관리** | **Stateful** (응답 트래픽 자동 허용) | **Stateless** (인/아웃바운드 각각 명시해야 함) |
| **규칙 성격** | **허용(Allow)만 가능** (화이트리스트) | **허용 및 거부(Deny) 둘 다 가능** |
| **적용 순서** | 모든 규칙을 종합 평가하여 적용 | 규칙 번호 순서대로(100, 200...) 순차 적용 |
| **대표 용도** | 웹 서버 포트 개방, 특정 IP 대역 허용 | 특정 악성 IP 주소 대역의 진입 차단 (Blacklist) |

## 제 8장. 거대한 성의 열쇠 체계, IAM (Identity and Access Management)

### 8.1 인증(Authentication) vs 인가(Authorization)
보안 그룹이 네트워크 통로를 지키는 보초병이라면, **IAM (Identity and Access Management)**은 "AWS 리소스를 생성하고 조작할 수 있는 관리자 권한"을 감시하는 거대한 자물쇠 체계입니다.

보안 공부의 기초는 이 두 개념을 분리하는 데서 시작합니다.
1. **인증 (Authentication - Who are you?)**: "당신이 주장하는 그 사람이 진짜 맞습니까?"를 검증하는 것. ID/PW 입력, 공인인증서 제출, MFA 토큰 입력 등이 여기에 속합니다.
2. **인가 (Authorization - What can you do?)**: "인증된 당신이 이 성 안에서 어떤 방(리소스)에 들어갈 수 있고, 어떤 보물(데이터)을 만질 수 있는가?" 권한을 검증하는 것입니다.

IAM은 AWS 웹 콘솔에 로그인하는 사람뿐 아니라, CLI나 API를 통해 "VPC를 생성해라", "EC2를 삭제해라"라고 요청하는 모든 행위자가 합당한 권한을 갖고 있는지 **인증하고 인가**하는 관제 타워입니다.

### 8.2 IAM의 4대 핵심 구성 요소
IAM을 잘 활용하려면 다음 4가지 핵심 명사를 완전히 지배해야 합니다.

1. **사용자 (User)**:
   - 실제 사람(개발자 김철수, 엔지니어 이영희) 또는 하나의 고유한 애플리케이션을 나타내는 물리적 계정입니다.
   - 고유한 로그인 비밀번호 또는 API 호출을 위한 Access Key / Secret Key 한 쌍을 발급받아 **인증**에 사용합니다.

2. **그룹 (Group)**:
   - 사용자들의 집합체입니다. (예: `Developer-Group`, `SysAdmin-Group`)
   - 매번 개개인에게 권한을 따로 주는 것은 비효율적이므로, 그룹에 권한 정책을 매달아 두고 사용자를 그룹에 쏙 집어넣어 권한을 일괄 상속시킵니다.

3. **역할 (Role)**:
   - **"사람이 아닌 리소스(예: EC2 서버)나 다른 계정이 임시로 뒤집어쓸 수 있는 가면"**입니다.
   - 예컨대 "내 EC2 인스턴스가 돌면서 S3 버킷에 로그 파일을 업로드해야 해"라고 할 때, EC2에 IAM 역할을 부여하면 EC2는 임시 보안 토큰을 발급받아 안전하게 권한을 수행합니다. 서버 하드디스크에 직접 위험한 Access Key를 저장하지 않게 해주는 강력한 보안 도구입니다.

4. **정책 (Policy)**:
   - **"무엇을 할 수 있고 무엇을 할 수 없는지"** JSON 형식의 문서로 촘촘히 적어둔 명세서입니다.
   - 이 정책을 사용자, 그룹, 역할에 '연결(Attach)'하여 인가를 수행합니다.

### 8.3 JSON 정책 문서 분석의 실제
IAM 정책은 다음과 같은 표준 JSON 스키마를 갖습니다. 아주 상세하게 뜯어봅시다.

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": [
        "ec2:RunInstances",
        "ec2:DescribeInstances",
        "ec2:StartInstances",
        "ec2:StopInstances"
      ],
      "Resource": "*",
      "Condition": {
        "StringEquals": {
          "aws:RequestedRegion": "ap-northeast-2"
        }
      }
    }
  ]
}
```

이 문서는 다음과 같이 한국어로 번역됩니다.
* **Effect (효과)**: `Allow` -> 허용하겠다! (또는 `Deny` -> 차단하겠다)
* **Action (작업)**: `ec2:RunInstances` 등 -> EC2 인스턴스를 생성하고, 조회하고, 켜고, 끄는 행동들을.
* **Resource (자원)**: `*` -> 모든 EC2 대상에 대해.
* **Condition (조건)**: `aws:RequestedRegion == ap-northeast-2` -> 단, 요청하는 리전이 **서울 리전(ap-northeast-2)**일 때만!

### 8.4 AWS 권한 평가 엔진의 내부 알고리즘
누군가 AWS API를 호출했을 때, AWS 내부의 권한 평가 시스템은 0.001초 만에 다음의 엄격한 **평가 로직 (Evaluation Logic)** 흐름도를 타고 의사결정을 내립니다.

```text
[AWS API 권한 평가 흐름]

                   시작 (API 요청 수신)
                           │
                           ▼
              1. 기본값으로 "거부" 설정 (Default Deny)
                           │
                           ▼
          2. 명시적 거부(Explicit Deny)가 정책 중에 있는가?
                     /           \
                  (Yes)          (No)
                   /               \
                  ▼                 ▼
          [ 즉 시 거 부 ]    3. 명시적 허용(Explicit Allow)이 정책 중에 있는가?
                                     /           \
                                  (Yes)          (No)
                                   /               \
                                  ▼                 ▼
                            [ 승 인 ]        [ 거 부 ]
```

1. **기본 거부 (Default Deny)**: 처음엔 무조건 거부 상태에서 시작합니다. 허용 규칙이 명시되어 있지 않다면 거부당합니다.
2. **명시적 거부 우선 (Explicit Deny Override)**: **이것이 제일 중요합니다.** 100개의 정책 중에 99개가 "너 이거 해도 돼(Allow)"라고 적어두었어도, 단 1개의 정책에 "너 이거 절대 안 돼(Explicit Deny)"라고 적혀 있다면, 그 요청은 **가차 없이 차단**됩니다. 거부(Deny) 권한은 허용(Allow) 권한보다 강력한 왕권을 가집니다.

### 8.5 최소 권한 원칙 (Principle of Least Privilege)
클라우드 초보자들이 가장 자주 범하는 최악의 실수 중 하나가 바로 귀찮다는 이유로 본인의 부하 직원이나 실습 사용자 계정에 **`AdministratorAccess` (관리자 권한)** 정책을 통째로 쥐여주는 것입니다. 

이것은 마치 집 열쇠를 만드는 게 귀찮아서 집 앞 비밀번호를 `0000`으로 뚫어두거나 지나가는 사람 모두에게 마스터키를 복사해 주는 것과 같습니다.

* **최소 권한 원칙**: "사용자에게 업무를 수행하는 데 필요한 **최소한의 권한 범위**만 부여하고, 그 외의 권한은 철저히 박탈한다."
* **AdministratorAccess의 위험성**: 
  이 권한을 가진 계정의 API 키가 한 번 유출되면, 공격자는 1초 만에 여러분의 모든 리소스를 삭제하고 계정을 영구 탈취한 뒤, 해당 계정으로 비트코인 채굴용 대형 EC2 인스턴스를 수백 대 돌려버립니다. 한 달 뒤 날아오는 **수천만 원에서 수억 원의 빌링 폭탄**은 고스란히 여러분의 몫이 됩니다.

이번 미션 요구사항 4.4절에서도 강조하듯, 우리는 `AdministratorAccess`를 절대 쓰지 않고, 오직 실습에 꼭 필요한 **EC2, VPC, Security Group 관련 액션들만 핀포인트로 조립한 커스텀 정책**을 사용해 최소 권한만을 할당하는 훈련을 하게 됩니다.

## 제 9장. 소켓 통신에서 컨테이너 가상화까지: Nginx와 Docker

### 9.1 웹 서버가 포트(Port)를 리스닝(Listening)한다는 것의 실체
네트워크 선을 타고 패킷이 드디어 내 EC2 인스턴스의 OS에 도달했습니다. 목적지 포트는 `80(HTTP)`입니다. 
컴퓨터 내부 커널은 이 패킷을 도대체 어떤 소프트웨어에게 전달해야 할까요?

이를 이해하려면 OS 커널의 **소켓(Socket)**과 **포트(Port)**의 개념을 알아야 합니다.
* **포트(Port)**: 비유하자면 1개의 IP 주소(아파트 주소) 내에 존재하는 65,536개의 가상 수신함(호수)입니다. 
* **소켓(Socket)**: OS 커널이 네트워크 통신을 위해 만들어주는 가상 인터페이스 파이프라인입니다.

웹 서버 소프트웨어(예: Nginx)가 서버에서 켜질 때, 커널에게 시스템 콜을 날려 **"앞으로 이 컴퓨터에 들어오는 포트 중 80번 포트로 들어오는 패킷은 몽땅 나한테 바인딩(Binding)해서 넘겨줘!"**라고 요구합니다. 이 상태를 **Listen(리스닝 - 귀를 기울이고 기다림)** 상태라고 합니다.

```text
[네트워크 패킷 유입]
목적지: 10.0.1.10:80  ──> [OS 커널 라우팅] ──> 80번 소켓 리스너 확인 ──> [Nginx 프로세스]
```

외부 사용자가 브라우저로 들어오면 커널은 80번 포트에 귀를 대고 있던 Nginx 프로세스의 소켓으로 데이터를 쏴주고, Nginx는 이를 해석해 우리가 설정한 HTML 파일이나 헬스체크 응답을 돌려보내는 것입니다.

### 9.2 Nginx의 경이로운 아키텍처: 비동기 이벤트 루프
웹 서버에는 아파치(Apache HTTP Server)와 엔진엑스(Nginx)라는 양대 산맥이 있습니다. 
아파치는 전통적으로 **Thread-per-connection(커넥션당 스레드 1개 생성)** 방식을 썼습니다. 손님이 100명 오면 일꾼(Thread)을 100명 고용해 각각 응답을 전담시키는 구조입니다. 이 방식은 손님이 수만 명으로 늘어나면 컴퓨터 내에서 스레드를 스위칭(Context Switching)하느라 정작 손님 응답은 못 하고 CPU가 과부하로 쓰러지는 한계가 있었습니다.

Nginx는 이 한계를 깨부수기 위해 **비동기 이벤트 드리븐 (Asynchronous Event-Driven)** 아키텍처로 설계되었습니다.

쉽게 말해, Nginx는 단 몇 명의 고정된 일꾼(Worker Process)만 고용합니다. 
그리고 손님들의 요청이 들어올 때마다 이를 하나의 거대한 **'이벤트 큐(Event Queue)'**에 휙휙 던져두고, 싱글 루프를 돌며 준비가 완료된 작업부터 비동기적으로 빛의 속도로 처리해 나갑니다. 

마치 베테랑 바텐더 한 명이 수십 명의 손님 주문을 동시에 조율하며 칵테일을 신속히 말아주는 것과 같습니다. 이 뛰어난 경량 아키텍처 덕분에 Nginx는 아주 적은 CPU/RAM 메모리만을 사용하고도 수만 개의 동시 접속자 요청을 가뿐히 소화해 내며, 현대 클라우드 인프라의 표준 웹 서버로 우뚝 서게 되었습니다.

### 9.3 컨테이너 가상화 vs VM 가상화의 근본적 차이 (Docker의 정수)
미션의 보너스 과제에는 **Docker(도커) 컨테이너**로 웹 서비스를 배포하는 멋진 요구사항이 포함되어 있습니다. 
이 단계에서 많은 초보자들이 "EC2 가상 컴퓨터 위에 또 Docker 가상 컴퓨터를 띄운다고? 이게 무슨 비효율이지?" 하고 헷갈려합니다.

여기서 **VM 가상화**와 **컨테이너 가상화(Docker)**의 패러다임 차이를 명확히 짚고 넘어가야 합니다.

```text
┌───────────────────────────────────────┐   ┌───────────────────────────────────────┐
│     가상 머신 (VM - 예: EC2)           │   │      컨테이너 (Container - 예: Docker)  │
├───────────────────────────────────────┤   ├───────────────────────────────────────┤
│  [App A]   [App B]                    │   │  [App A]   [App B]                    │
│  [Lib A]   [Lib B]                    │   │  [Lib A]   [Lib B]                    │
│ ┌───────────────────────────────────┐ │   │ ┌───────────────────────────────────┐ │
│ │  Guest OS (Kernel 포함)           │ │   │ │ Docker Engine (Virtual OS 없음)   │ │
│ └───────────────────────────────────┘ │   │ └───────────────────────────────────┘ │
│ ┌───────────────────────────────────┐ │   │ ┌───────────────────────────────────┐ │
│ │ Hypervisor 가상화 엔진             │ │   │ │ Host OS Kernel (공유하여 사용)     │ │
│ └───────────────────────────────────┘ │   │ └───────────────────────────────────┘ │
│ ┌───────────────────────────────────┐ │   │ ┌───────────────────────────────────┐ │
│ │ Host Hardware (물리 인프라)        │ │   │ │ Host Hardware (물리 인프라)        │ │
│ └───────────────────────────────────┘ │   │ └───────────────────────────────────┘ │
└───────────────────────────────────────┘   └───────────────────────────────────────┘
```

1. **VM 가상화 (하드웨어 가상화)**:
   - 가상 컴퓨터 위에 **또 하나의 완전한 운영체제(Guest OS)**를 설치합니다.
   - OS 커널 부팅 프로세스부터 메모리 관리 시스템까지 통째로 가상화하므로, 독자적인 격리성은 뛰어나지만 수백 MB~수 GB의 메모리 낭비가 발생하고 기동 시간이 수 분으로 느립니다.

2. **컨테이너 가상화 (운영체제 수준 가상화 - Docker)**:
   - 가상 OS를 새로 설치하지 않습니다! 대신 호스트 컴퓨터의 **리눅스 커널(Kernel)을 그대로 공유**합니다.
   - 단지 리눅스 커널 내부의 강력한 격리 기술인 **네임스페이스(Namespaces - 프로세스 ID, 네트워크, 사용자 등을 격리)**와 **cgroups(Control Groups - CPU, 메모리 자원 사용량 제한)**라는 기능을 활용해, 특정 프로세스가 마치 독자적인 OS 환경에 갇혀있는 것처럼 눈속임만 해줍니다.
   - 가상 OS를 부팅하는 비효율이 없으므로, Docker 컨테이너를 켜는 데 드는 시간은 단 **0.1초**에 불과하며, 메모리 오버헤드도 사실상 0에 가깝습니다.

Docker 덕분에 우리는 "어느 컴퓨터 환경(내 노트북이든, AWS EC2 환경이든)에서든 정확히 똑같이 실행되는 격리된 애플리케이션 실행 환경"을 극도로 가볍고 빠르게 빌드하고 배포할 수 있는 혁명을 맞이하게 되었습니다.

## 제 10장. 프로페셔널의 기술: 트러블슈팅(Troubleshooting) 방법론

### 10.1 엔지니어의 핵심 경쟁력: 오류를 해결하는 자세
시니어 인프라 엔지니어와 주니어 인프라 엔지니어를 가르는 단 하나의 차이는 **"오류를 만났을 때 어떻게 반응하는가"**입니다.
* 주니어: "어? 왜 안 되지?" 하며 설정창을 이것저것 무작위로 클릭해 보거나 웹캠처럼 화면만 쳐다봅니다.
* 시니어: 즉시 터미널 로그를 열고, 네트워크 흐름도를 그리며 **가설을 세우고 논리적으로 검증**해 나갑니다.

미션의 필수 결과물인 **트러블슈팅 보고서**는 바로 이 "엔지니어의 논리적 문제 해결 사고 흐름"을 평가하는 가장 강력한 척도입니다.

### 10.2 대표적 에러 증상과 해결을 위한 가설 수립 및 검증 방법
클라우드 인프라 구축 시 발생하는 대표적인 3대 장애 상황에 대한 마스터 키 가이드를 제공합니다.

#### 시나리오 A: SSH 접속 시 "Connection Timeout" (접속 시간 초과)
* **증상**: `ssh -i ...` 명령을 치고 오래 기다렸는데 `ssh: connect to host x.x.x.x port 22: Connection timed out` 에러가 나며 뚝 끊깁니다.
* **컴퓨터 과학적 분석**: 
  `Timeout`은 패킷이 목적지 컴퓨터(EC2) 근처까지는 가려 했으나, 네트워크 벽(방화벽)에서 대답조차 듣지 못하고 가로막혀 조용히 증발(Drop)했음을 뜻합니다.
* **논리적 가설 및 검증 로직**:
  1. **가설 1**: 보안 그룹(SG) 인바운드 규칙에 22번 포트가 막혀 있다.
     - *검증*: AWS 콘솔에서 보안 그룹의 인바운드 소스가 내 현재 퍼블릭 IP와 정확히 일치하는지 확인합니다. (카페 등에서 일할 때 내 IP가 바뀌는 경우가 대다수입니다!)
  2. **가설 2**: 서브넷에 인터넷 게이트웨이(IGW)로 향하는 라우팅 테이블이 누락되었다.
     - *검증*: EC2가 있는 서브넷의 라우팅 테이블에 `0.0.0.0/0 -> igw-xxxx` 규칙이 등록되어 있는지 확인합니다.
  3. **가설 3**: EC2 인스턴스가 퍼블릭 IP를 아예 부여받지 못했다.
     - *검증*: 콘솔에서 인스턴스의 '퍼블릭 IPv4 주소' 칸이 비어있는지 확인합니다.

#### 시나리오 B: 웹 사이트 접속 시 "Connection Refused" (연결 거부)
* **증상**: 브라우저나 터미널에서 접속하려는데 `curl: (7) Failed to connect to x.x.x.x port 80: Connection refused` 에러가 즉시 떨어집니다.
* **컴퓨터 과학적 분석**: 
  `Refused`는 패킷이 보안 그룹 방화벽을 뚫고 **EC2 인스턴스 운영체제 안까지 성공적으로 들어갔음**을 의미합니다! 그러나 OS 커널이 패킷을 수신해 포트 대장을 열어봤더니, 80번 포트에 리스닝하고 있는 소프트웨어(Nginx)가 없어서 커널이 패킷을 거절(RST 패킷 응답)한 상황입니다.
* **논리적 가설 및 검증 로직**:
  1. **가설 1**: Nginx 웹 서버 프로그램이 설치되지 않았거나 꺼져 있다.
     - *검증*: SSH로 서버에 들어가 `sudo systemctl status nginx`를 쳐서 Nginx가 `active (running)` 상태인지 확인합니다.
  2. **가설 2**: Nginx가 80번 포트가 아닌 엉뚱한 포트(예: 8080)로 돌아가고 있다.
     - *검증*: 서버 내에서 `sudo netstat -tpln` 또는 `sudo ss -tpln`을 입력해 어떤 포트들이 리스닝 상태인지 확인합니다.

#### 시나리오 C: AWS CLI 명령어 사용 시 "Access Denied" (권한 거절)
* **증상**: AWS CLI나 인프라 관리 도구를 통해 리소스를 조회하려는데 `An error occurred (AccessDenied) when calling the DescribeInstances operation` 에러가 납니다.
* **컴퓨터 과학적 분석**: 
  내 API 요청이 AWS IAM 인증 관제탑에 도달해 인증은 통과했으나, 나에게 연결된 **IAM 정책(Policy)** 중에 해당 작업을 수행할 수 있는 권한(`Allow`)이 부여되지 않은 상태입니다.
* **논리적 가설 및 검증 로직**:
  1. **가설 1**: 내가 로그인한 IAM 사용자 계정에 필요한 권한 정책이 붙어있지 않다.
     - *검증*: IAM 콘솔로 가서 해당 사용자의 정책 리스트에 `AmazonEC2ReadOnlyAccess` 또는 커스텀 EC2 조회 정책이 들어있는지 확인합니다.
  2. **가설 2**: 명시적 거부(Deny) 정책이 어딘가 작동해 권한을 씹어버리고 있다.
     - *검증*: 여러 권한 정책 중 `Effect: Deny`가 섞여 있는지 혹은 IAM 권한 경계(Permissions Boundary)에 묶여 있는지 확인합니다.

## 제 11장. 리소스의 생애주기 관리와 비용 최적화 (Cleanup)

### 11.1 클라우드의 양날의 검: 종량제 과금의 경고
클라우드는 혁신적입니다. 하지만 빌려 쓰는 모델이기에 **"삭제하는 것을 잊은 자원"은 조용히 숨 쉬며 여러분의 통장에서 매시간 돈을 훔쳐 갑니다.** 

실습용 계정이나 회사 샌드박스 계정에서 자원을 방치했다가 한 달 뒤 엄청난 고지서를 받고 눈물을 흘리는 일은 클라우드 업계에서 매일 일어나는 잔혹동화입니다. 

### 11.2 삭제 시 꼬여버리는 자원 의존성 체인과 해법
AWS 리소스를 지우려고 할 때 가장 우리를 짜증 나게 만드는 에러 메시지가 있습니다.
> `VPC contains active resources and cannot be deleted.` (VPC 내에 활성 리소스가 남아있어 삭제할 수 없습니다)

이 현상이 발생하는 이유는 AWS 가상 리소스들 간에 **종속성 체인(Dependency Chain)**이 얽혀있기 때문입니다. 예컨대 아래와 같은 먹이사슬 구조입니다.

```text
[AWS 리소스 의존성 먹이사슬]
VPC ──> Subnet ──> 네트워크 카드(ENI) ──> EC2 인스턴스 ──> EBS 볼륨
  │                                           │
  └──> Internet Gateway                       └──> Elastic IP (고정 공인 IP)
```

이 사슬 구조에서 부모(VPC)를 먼저 지우려 하면, 자식(서브넷, EC2)이 살아서 펄떡이고 있으므로 커널 수준에서 삭제를 강력히 거부합니다. 따라서 우리는 아래와 같이 **정확히 자식부터 부모 순으로 거꾸로 타고 올라가며 해체하는 순서**를 숙지해야 합니다.

### 11.3 마스터 리소스 정리(Cleanup) 5단계 프로토콜
실습이 끝나면 무조건 자다가도 깨어나서 이 공식대로 리소스를 하나씩 파괴해 나가십시오.

1. **1단계: EC2 인스턴스 종료 (Terminate)**
   - 인스턴스를 '중지(Stop)'하는 데 그치면 안 됩니다. 중지된 EC2는 CPU 비용은 안 나가지만, **EBS 하드디스크 볼륨 요금은 계속 청구**됩니다. 완전히 지구상에서 지워버리는 **'종료(Terminate)'**를 눌러야 합니다. 
   - 인스턴스를 종료하면 기본적으로 같이 붙어있던 메인 EBS 볼륨도 함께 세트로 삭제됩니다.

2. **2단계: 탄력적 IP (Elastic IP, EIP) 해제 (Release)**
   - **[가장 무서운 함정]**: Elastic IP(고정 공인 IP)는 내 EC2 서버에 연결되어 사용 중일 때는 요금이 무료입니다. 하지만 **EC2를 지워버리고 EIP만 덩그러니 남겨두면, AWS는 즉시 매시간당 벌금 성격의 유휴 요금을 과금하기 시작합니다!** (한정 자원인 공인 IP를 쓰지도 않으면서 선점해 두고 방치했다는 벌금입니다)
   - 따라서 EC2를 종료한 뒤에는 반드시 EIP 메뉴로 가서 **'탄력적 IP 주소 릴리스(Release)'**를 눌러 우주로 돌려보내 주어야 합니다.

3. **3단계: 인터넷 게이트웨이 (IGW) 분리 및 삭제**
   - 인터넷 게이트웨이가 VPC에 붙어(Attach) 있는 동안에는 VPC를 지울 수 없습니다. 
   - 먼저 **'VPC에서 분리(Detach)'**를 실행한 뒤, 비로소 **'삭제(Delete)'**를 단행합니다.

4. **4단계: 보안 그룹 및 서브넷, VPC 삭제**
   - 이제 장애물이 모두 제거되었습니다. VPC 메뉴로 이동해 해당 VPC를 통째로 선택하고 **'VPC 삭제'**를 누르면, 그 밑에 딸려 있던 가상 서브넷들과 라우팅 테이블들이 연쇄적으로 깔끔하게 일괄 폭파됩니다.

5. **5단계: 빌링 대시보드(Billing Dashboard) 최종 교차 검증**
   - 리소스를 다 지웠다고 안심하지 마십시오. 하루 정도 지난 뒤 AWS **결제 및 비용 관리(Billing Dashboard)** 메뉴로 이동하여 내 청구 금액과 활성 리소스 내역에 1센트라도 움직임이 생기고 있는지 두 눈으로 직접 확인하는 습관이 클라우드 프로페셔널의 마지막 자격입니다.

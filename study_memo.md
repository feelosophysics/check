## 주의사항

#### 요금 
- 인스턴스 스탑

- 중지(Stop)와 종료(Terminate)의 차이:
    - 중지(Stop): 컴퓨터 전원만 끈 상태입니다. CPU 요금은 안 나가지만, 하드디스크(EBS 볼륨)는 남아있어 스토리지 요금이 발생할 수 있습니다.
    - 종료(Terminate): 컴퓨터를 아예 폐기하는 것입니다. 과제를 마치고 리소스를 정리할 때는 반드시 '종료'를 선택해야 합니다.

- 탄력적 IP (Elastic IP, EIP)의 함정: 가장 많이 과금되는 요인 중 하나입니다. EIP는 실행 중인 EC2 인스턴스에 연결되어 있을 때는 무료지만, 인스턴스를 중지(Stop)하거나 삭제했는데 EIP를 반납(Release)하지 않고 쥐고 있으면 시간당 요금이 청구됩니다. 과제를 마치면 반드시 EIP부터 '릴리스(해제)' 하세요.

- 결제 알림(Billing Alarm) 설정: 실습 시작 전 AWS Budgets 서비스나 결제 대시보드에 들어가서 "예상 청구액이 $1를 초과하면 이메일 알림"을 보내도록 가장 먼저 설정하세요.

#### 보안

- 루트(Root) 계정 봉인: 이메일 주소로 로그인하는 루트 계정은 무한대의 권한을 가집니다. 과제 명세서대로 실습용 IAM 사용자를 따로 만들고, 그 사용자에게 필요한 권한(EC2, VPC 등)만 부여하여 로그인(IAM 사용자 로그인) 하세요.

- Key Pair 파일(.pem) 절대 사수: EC2에 접속하기 위해 발급받는 키 페어 파일은 단 한 번만 다운로드할 수 있으며, 비밀번호와 같습니다.
절대 GitHub 같은 퍼블릭 저장소에 코드를 올릴 때 함께 올리지 마세요. 해커들이 봇으로 1분 만에 키를 긁어가서 전 세계 리전에 수백 대의 고성능 서버를 띄워 암호화폐를 채굴합니다(수천만 원 과금 발생).

- 보안 그룹(Security Group) 규칙: 과제 요구사항에 있는 "SSH(22)는 학습자 개인 IP에서만 접근 가능하게 하라"는 필수입니다. 소스(Source)를 내 IP(My IP)로 설정하세요. 0.0.0.0/0(모두 허용)으로 열어두면 전 세계의 해커 봇들이 끊임없이 접속(해킹) 시도를 합니다.

#### 인프라 체크
- 리전(Region) 확인: AWS 콘솔 우측 상단에 지역이 서울(ap-northeast-2) 로 되어 있는지 항상 확인하세요. 나도 모르게 미국(버지니아 북부 등)에 자원을 만들면 나중에 찾지도 못하고 요금만 나옵니다.

- 인터넷 게이트웨이(IGW)와 라우팅 테이블(Route Table): VPC와 서브넷을 만들고 IGW를 붙였다고 끝이 아닙니다. 반드시 서브넷의 '라우팅 테이블'에 가서 0.0.0.0/0 목적지를 만들어 방금 만든 IGW로 향하도록 길을 뚫어주어야 인터넷이 통신됩니다. (초보자가 가장 많이 헤매는 트러블슈팅 포인트 1순위입니다.)

- 퍼블릭 IP 자동 할당: EC2를 생성할 때 네트워크 설정에서 '퍼블릭 IP 자동 할당'이 활성화(Enable) 되어 있어야 외부에서 브라우저로 접속할 수 있는 공인 IP가 부여됩니다.

### 자원 삭제(Cleanup) 순서
- AWS는 만들 때의 역순으로 지워야 깔끔하게 지워집니다. 안 그러면 "의존성 때문에 삭제할 수 없습니다"라는 에러가 뜹니다.
1. EC2 인스턴스 종료(Terminate)
2. (할당했다면) 탄력적 IP(EIP) 연결 해제 후 릴리스(Release)
3. VPC 메뉴로 이동하여 Internet Gateway를 VPC에서 분리(Detach) 후 삭제
4. Subnet 삭제 및 Route Table 삭제
5. 최종적으로 VPC 삭제

## 4. 기능 요구 사항
다음 요구사항을 모두 만족해야 한다.

#### 4.1 네트워크 구성
  - VPC 1개를 생성한다.
  - Public Subnet 1개를 생성한다.
  - Internet Gateway를 VPC에 연결한다.
  - Public Subnet의 Route Table에 0.0.0.0/0 → Internet Gateway 경로가 존재해야 한다.
  - Public Subnet에 배치된 인스턴스는 인터넷으로 아웃바운드 통신이 가능해야 한다. (예: curl <https://example.com> 성공)
#### 4.2 컴퓨트 및 웹 서버 배포
  - Public Subnet에 EC2 인스턴스 1대를 생성한다.
  - SSH로 인스턴스에 접속할 수 있어야 한다.
  - 웹 서버(Nginx 등)를 설치하고 실행 상태여야 한다.
  - 인스턴스에서 curl <http://localhost> 요청이 200 응답을 반환해야 한다.
#### 4.3 접근 제어(보안 그룹)
  - Security Group 인바운드는 원칙적으로 필요한 포트만 허용한다.
  - HTTP(80)는 0.0.0.0/0에서 접근 가능해야 한다.
  - SSH(22)는 학습자 개인 IP(또는 지정된 IP 대역)에서만 접근 가능해야 한다.
  - 0.0.0.0/0에 대해 전체 포트(0-65535) 허용 규칙은 생성하지 않는다.
#### 4.4 권한(IAM) 최소권한
  - 실습에 사용하는 IAM 사용자(또는 Role) 1개를 사용한다.
  - 해당 IAM 권한은 EC2/VPC/Security Group 구성에 필요한 범위로 제한한다.
    - 예: 생성·조회·태그·연결 등 실습에 필요한 작업 중심
    - 예: 실습과 무관한 서비스(S3/RDS 등) 권한은 부여하지 않음
  - 관리자 권한(AdministratorAccess)은 부여하지 않는다.
#### 4.5 외부 접속 검증(택 1)
  - 아래 방식 중 1개를 선택해 “외부에서 접속 가능”을 증명해야 한다.

1. VPC, Subnet, Route Table, Internet Gateway의 역할 및 트래픽 흐름
🌐 핵심 구성 요소의 역할
VPC (Virtual Private Cloud)
역할: AWS 클라우드 내에 정의된 고객만의 논리적으로 격리된 가상 네트워크 공간(영토)입니다. 다른 사용자들로부터 완벽하게 분리되어 독립된 네트워크 환경을 보장합니다.
Subnet (서브넷)
역할: VPC라는 큰 네트워크 영토를 용도나 보안 수준에 따라 더 세분화하여 쪼갠 구역(방)입니다. 인터넷과 양방향 통신이 가능한 **퍼블릭 서브넷(Public Subnet)**과 내부망으로만 쓰이는 **프라이빗 서브넷(Private Subnet)**으로 나뉩니다.
Route Table (라우팅 테이블)
역할: 서브넷 내부에서 발생하는 트래픽이 목적지(IP 대역)를 찾아갈 수 있도록 길을 안내해 주는 이정표(길 안내자)입니다.
Internet Gateway (Internet Gateway, IGW)
역할: VPC와 외부 인터넷 세상을 연결해 주는 유일한 대문(관문)입니다. 사설 IP(Private IP)와 공인 IP(Public IP) 간의 1:1 NAT 변환을 처리하여 통신을 가능케 합니다.
🛣️ 트러블슈팅을 위한 트래픽 흐름 (외부 인터넷 ➔ EC2 인스턴스)
외부 사용자(Client)가 웹 브라우저를 통해 접속을 시도하면 트래픽이 VPC의 관문인 **인터넷 게이트웨이(IGW)**에 도착합니다.
IGW는 유입된 트래픽을 VPC 내부로 전달하며, 해당 서브넷에 연결된 **라우팅 테이블(Route Table)**의 안내를 받습니다.
라우팅 테이블의 규칙(0.0.0.0/0 -> IGW)에 따라 트래픽이 해당 **퍼블릭 서브넷(Public Subnet)**의 목적지 주소로 향합니다.
퍼블릭 서브넷 경계에 진입한 트래픽은 EC2 인스턴스를 감싸고 있는 가상 방화벽인 **보안 그룹(Security Group)**의 인바운드 규칙을 확인합니다.
허용된 포트(예: HTTP 80, HTTPS 443) 규칙을 무사히 통과하면 최종적으로 **EC2 인스턴스 내의 웹 서버(Nginx)**에 도달하여 응답 페이지를 반환합니다.
2. 외부 요청이 EC2의 웹 서버까지 도달하기 위해 필요한 설정
외부 브라우저의 웹 요청이 퍼블릭 서브넷에 배치된 EC2 Nginx 웹 서버까지 정상 도달하려면 다음 3가지 핵심 설정이 유기적으로 맞물려야 합니다.

퍼블릭 IP (Public IP) 할당
논리적 이유: VPC 내부에서 사용하는 IP는 외부에서 접근할 수 없는 사설 IP(Private IP, 예: 10.0.1.x)입니다. 따라서 외부 인터넷 사용자가 브라우저를 통해 통신을 개시할 수 있도록 전 세계에서 고유한 공인 IP 주소(퍼블릭 IP)를 EC2 인스턴스에 반드시 할당해야 합니다.
인터넷 게이트웨이(IGW) 경로 설정 (라우팅)
논리적 이유: 단순히 대문(IGW)을 만드는 것뿐만 아니라, 서브넷에서 외부로 나가는 길을 뚫어주어야 합니다. 퍼블릭 서브넷에 지정된 라우팅 테이블에 **"로컬 VPC 대역을 제외한 모든 외부 인터넷(0.0.0.0/0)으로 향하는 트래픽은 인터넷 게이트웨이(IGW)를 통해라"**는 라우팅 경로가 지정되어 있어야 외부 유입 및 응답 반환 통신이 정상적으로 동작합니다.
보안 그룹 (Security Group) 인바운드 규칙 허용
논리적 이유: AWS의 보안 그룹은 Statefull 방화벽으로, 들어오는 트래픽(인바운드)을 원천 차단하고 허용된 규칙만 통과시킵니다. 따라서 웹 서버로 들어오는 HTTP 서비스 포트인 **80번 포트(및 HTTPS 443번 포트)**에 대해 소스 0.0.0.0/0(모든 IP 대상)으로 허용 규칙이 생성되어 있어야 최종적으로 서버 내부 포트까지 트래픽이 도달할 수 있습니다.
3. 클라우드 과금이 발생하는 대표 요인 및 안전한 리소스 정리 순서
💸 대표적인 과금 요인
EC2 인스턴스 중지(Stop) 상태 방치
인스턴스를 중지하면 컴퓨터 전원만 꺼질 뿐, 저장 장치인 EBS(Elastic Block Store) 볼륨은 그대로 남아있기 때문에 스토리지 보관 요금이 계속 청구됩니다.
미연결 탄력적 IP (Elastic IP, EIP) 방치
탄력적 IP는 한정된 공인 IP 자원이므로, 인스턴스에 연결되어 작동 중일 때는 무료이나 연결된 인스턴스가 없거나 꺼져 있는 상태로 유지되면 점유 시간당 페널티 요금이 과금됩니다.
NAT Gateway, ALB, RDS 등 고단가 리소스 삭제 누락
실습 도중 임시로 생성한 로드 밸런서(ALB)나 데이터베이스(RDS), 프라이빗 통신을 위한 NAT Gateway 등은 시간당 기본 요금이 매우 비싼 편이라 방치 시 큰 금액이 청구됩니다.
🔄 안전한 리소스 정리 순서와 이유
AWS 인프라는 자원 간의 긴밀한 의존 관계(VPC ➔ Subnet ➔ EC2 ➔ Security Group 등)가 얽혀 있습니다. 겉의 네트워크 껍데기(VPC)를 먼저 지우려 하면 안의 내부 자원(EC2) 때문에 의존성 에러가 발생하므로, 반드시 **"안에 들어있는 내용물(EC2)을 먼저 제거한 후 껍데기를 해체하는 역순"**으로 정리해야 합니다.

EC2 인스턴스 종료 (Terminate)
이유: 컴퓨터 본체를 영구 폐기하고, 함께 장착된 EBS 볼륨도 기본 설정에 의해 자동으로 함께 삭제되도록 유도합니다. (상태가 완전히 Terminated 될 때까지 1-2분 대기해야 관련 종속성이 풀립니다.)
탄력적 IP (Elastic IP) 릴리스 (Release)
이유: 인스턴스가 종료되어 연결이 끊어진 상태이므로, EIP를 즉시 반납하여 미연결 IP 자원 점유 과금을 차단합니다.
보안 그룹 (Security Group) 및 키 페어 삭제
이유: EC2 인스턴스에 적용되어 보안 규칙을 담당하던 그룹을 안전하게 폐기합니다.
VPC 삭제
이유: 가장 바깥의 영토인 VPC를 최종적으로 삭제합니다. VPC를 삭제하면 그 안에 속해 있던 서브넷(Subnet), 인터넷 게이트웨이(IGW), 라우팅 테이블(Route Table) 등이 삭제 오류 없이 한 번에 묶여서 안전하게 일괄 자동 삭제됩니다.














## 1. 클라우드와 인프라의 여정: 물리 서버에서 가상화까지

- on-premise: 기업이나 개인이 클라우드 같은 원격 서비스(제3자의 서버)를 빌려 쓰지 않고, 자체적으로 전산실이나 데이터 센터 등 직접 물리적 서버와 IT 인프라를 보유하고 설치하여 운영하는 방식

- hypervisor: 물리 하드웨어 위에 올라가 가상 컴퓨터를 창조하고 통제하는 소프트웨어 레이어.
    - 하드웨어(CPU, Memory, Disk, Network)를 논리적으로 잘게 쪼개어 가상 머신(Virtual Machine, VM)에 나누어 줍니다. VM 내부의 게스트 OS(Guest OS)는 자신이 진짜 물리 컴퓨터 위에서 독점적으로 돌고 있다고 굳게 믿지만, 실제로는 하이퍼바이저가 분할해 준 일부의 자원만을 쓰고 있는 것


- SDN (Software Defined Networking), 네트워크 가상화: 패킷의 헤더에 가상 네트워크 식별자를 붙이고 가상 스위치가 이를 소프트웨어적으로 라우팅 및 캡슐화(Encapsulation)하여 배달하는 방식을 사용.
    - 이러한 고도의 가상 네트워크 기술 위에 구현된, AWS 고객별 독립적인 가상 네트워크 공간이 바로 **VPC (Virtual Private Cloud)**
        - VPC는 쉽게 말해 **"AWS라는 거대한 가상의 우주 안에 나만을 위해 만들어진 독점적이고 고립된 영토"**입니다. 내 VPC 안에서 돌아가는 서버들은 다른 고객의 VPC 내 서버들과 철저히 격리되어, 서로 어떠한 트래픽도 주고받을 수 없는 완벽한 보안 환경을 보장받습니다.

- 서브넷: 이(브로드캐스트 도메인 비대화, 보안) 때문에 우리는 거대한 VPC 영토 안에 **"보안 벽과 파티션(라우팅 경계)"**을 쳐서 네트워크 영역을 쪼개게 되는데, 이 쪼개진 방들을 **서브넷 (Subnet, 하위 네트워크)**이라고 부릅니다.

- 퍼블릭 서브넷
- AWS 서브넷에서 사라진 5개의 예약 IP 주소

- 로컬 라우팅(Local Route) 규칙: VPC 내부에서 서브넷을 만들면, 그 서브넷의 라우팅 테이블에는 아무 짓도 하지 않아도 무조건 가장 먼저 등록되어 있고 절대 삭제할 수 없는 절대적인 규칙

- 세상과 소통하는 관문, Internet Gateway (IGW)
    - 외부 통로 제공: VPC 내부의 자원들이 바깥 인터넷에 접속하고, 바깥 인터넷 사용자들도 VPC 내부로 유입될 수 있는 관문이 됩니다.
    - 1:1 Static NAT (Network Address Translation) 처리: 이것이 바로 IGW의 핵심 마법입니다.
    - WS의 인터넷 게이트웨이는 이름만 '게이트웨이'일 뿐, 창고에 서 있는 단일 물리 장비나 가상 머신이 아닙니다. 사실 IGW는 AWS 소프트웨어 정의 네트워크(SDN) 상에 널리 분산되어 논리적으로 동작하는 고가용성, 초확장성 가상 커넥터입니다.
    - 실제 트래픽은 데이터 센터 내의 수많은 물리 스위치 노드들을 통해 고도로 분산되어 처리되므로, 대역폭의 한계가 없고 하드웨어 결함으로 인해 IGW가 죽는 일 또한 구조적으로 일어나지 않습니다. AWS가 우리에게 제공하는 완벽한 가상화 추상화의 이점입니다.

- EBS (Elastic Block Store): 가상 컴퓨터의 하드디스크

- Nginx의 구조적 정의
    - Nginx는 비동기 이벤트 기반(Event-Driven) 구조를 가진 웹 서버 소프트웨어입니다. 클라이언트(브라우저)의 요청을 받아 정적 콘텐츠를 응답하거나, 뒤에 있는 애플리케이션 서버로 요청을 전달하는 역할을 수행합니다.
    - 구조적 위치: 사용자(Client)와 백엔드 서버(Application Server, 예: Node.js, Python, Java) 사이의 **'앞단'**에 위치하여 게이트웨이 역할을 합니다.
    - 작동 방식: 과거의 웹 서버(Apache 등)가 요청 하나당 프로세스나 스레드를 하나씩 할당하여 자원을 많이 소모했던 것과 달리, Nginx는 적은 수의 프로세스로 수만 개의 연결을 동시에 처리하는 비동기 방식을 채택하고 있습니다.

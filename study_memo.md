## 3.5 시스템 무응답(Freeze) 상태 진단을 위한 3대 핵심 리눅스 표준 유틸리티 활용법
운영 환경에서 서비스가 갑자기 무응답 프리징 상태에 도달했을 때, 원인이 데드락인지 단순 대기인지 근거를 수집하는 실전 트러블슈팅 3단계 진단법을 계통화합니다.

```
  [ 무응답 감지 ] ➔ 1단계: ps -ef ➔ PID 생존 확인 (살아있는 좀비인가?)
                         │
                         v
                  2단계: ps -L / top -H ➔ 스레드 리소스 정체 증명 (스레드가 잠들었는가?)
                         │
                         v
                  3단계: pstack / gdb ➔ 락 대기 코드 지점 검증 (어떤 자원을 기다리는가?)
```

### ① 1단계: PID 생존 검증 (`ps -ef | grep agent`)
* **목적**: 프로그램이 완전히 죽은 크래시인지, 아니면 영혼 없이 몸뚱이만 살아있는 데드락 프리징 상태인지 구별합니다.
* **진단 결과**: 프로세스 ID가 활발히 조회된다면, 프로세스가 메모리에 적재되어 OS 커널 관리 하에 프로세스 테이블에는 등록되어 있으나 연산을 전혀 안 하고 멈춰있는 상태임을 1차 확보합니다.

### ② 2단계: 스레드 자원 점유 상태 정밀 감시 (`ps -L` 및 `top -H`)
* **명령어**: `ps -L -p [PID]` 혹은 `top -H -p [PID]`
* **기저 원리**: `-L`과 `-H` 옵션은 프로세스 내부의 모든 경량 스레드(LWP)들을 펼쳐서 보여줍니다.
* **진단 결과**: 스레드별 CPU 사용 시간(`TIME`) 필드를 수십 초 간격으로 지켜봅니다. 사용 시간이 전혀 누적되지 않고 특정 밀리초 대역에서 소수점 끝자리 하나 바뀌지 않고 정체되어 있다면, 해당 스레드가 사용자 연산을 전혀 하지 않고 커널 레벨에서 락 신호를 무기한 대기하며 잠들어 있는(Sleeping, `S` 상태) 결정적 물증이 됩니다.

### ③ 3단계: 스레드 스택 콜트레이스 역추적 (`pstack` 및 `gdb`)
* **명령어**: `pstack [PID]` 혹은 `gdb -p [PID] -ex "thread apply all bt"`
* **기저 원리**: 현재 실행 중인 모든 스레드의 함수 호출 스택 프레임을 거꾸로 추적해 현재 머물러 있는 코드 라인을 출력합니다.
* **진단 결과**: 스레드 A가 `__lll_lock_wait` 또는 `pthread_mutex_lock` 함수 프레임에 걸려 멈춰있고, 스레드 B 또한 동일한 동기화 락 대기 지점에서 멈춰 있음을 실시간 스택 메모리 상에서 육안으로 확인하여 데드락을 100% 물리적으로 입증합니다.

## memo
- 메모리 힙(Heap) 영역: 컴퓨터 프로그래밍에서 데이터의 **동적 할당(Dynamic Allocation)**을 위해 사용되는 핵심적인 메모리 공간입니다. 프로그램이 실행되는 도중(Runtime)에 크기가 결정되는 데이터를 저장하며, 개발자가 직접 제어하거나 언어의 런타임 환경(Garbage Collector)이 관리하는 영역
- Stack: 컴파일 타임에 할당, 관리 방식: OS/컴파일러(자동), heap보다 매우 빠름, 크기는 제한적(보통 작음), 데이터 구조는 LIFO

- tcmalloc(Thread-Caching Malloc): google에서 개발한 고성능 메모리 할당기
    - Thread Cache: 각 스레드마다 작은 메모리 공간을 미리 할당해 둡니다. 스레드가 메모리를 요청하면 전역 락 없이 자신의 캐시에서 즉시 가져옵니다.
- jemalloc(Jason Evans' Malloc): FreeBSD, Facebook(Meta), Rust 등에서 기본 혹은 선택적으로 사용하는 할당기
    - 메모리 단편화(Fragmentation) 최소화와 멀티코어 확장성(Scalability) 극대화에 초점
    - Arenas: 힙을 여러 개의 '아레나(Arena)'로 나눔. 스레드들은 각자 지정된 아레나를 사용하여 락 경합을 분산시킴.
- Object Pool: 메모리 할당/해제 자체를 시스템에 맡기지 않고 애플리케이션 레벨에서 직접 관리하는 디자인 패턴.
- Data-Oriented Design(DOD, 데이터 지향 설계): 객체 지향 설계(OOP)의 한계를 극복하기 위해 등장한 시스템 아키텍처 철학
    - 핵심 철학: "데이터가 위치한 방식이 성능을 결정한다."

- FCFS: First-Come, First-Served, 단순한 CPU 스케줄링 알고리즘. 자료구조의 Queue(FIFO)와 동일한 논리로 작동.
    - 단점: 호위 효과Convoy Effect

- Memory Page
    - Page: 가상 메모리 관리 기법 Paging에서 사용하는 **고정 크기의 최소 메모리 블록**
    - 외부 단편화External Fragmentation 해결하기 위해.
    - Page Fault 이슈: 프로세스가 접근하려는 페이지가 현재 RAM에 없을 때 발생하는 인터럽트. OS는 이를 감지하여 디스크에서 해당 데이터를 찾아 RAM의 빈 프레임에 올립니다. 이 과정이 너무 빈번하면 성능이 급격히 떨어지는 **쓰레싱(Thrashing)** 현상이 발생
- MMU(Memory Management Unit): 가상 주소를 실제 RAM의 물리 주소로 변역하는 고속 하드웨어 변환 장치. CPU 옆에 위치.
    - 가상 메모리를 일정한 단위인 page로 쪼개고, 이에 매핑되는 물리 메모리의 단위인 frame을 매칭하는 주도면밀한 가교 역할을 수행. 현재 대부분의 시스템에서 표준 페이지 크기는 4KB.
- TLB(Translation Lookaside Buffer): 자주 번역되는 페이지 주소 번역 쌍을 캐싱해 두는 초고속 하드웨어 캐시 메모리.
- VSZ(Virtual Memory Size)
    - 정의: 프로세스가 실행되면서 확보한 가상 메모리 전체 공간의 총합.
    - 특성: 코드 영역, 데이터 영역, 힙, 스택뿐만 아니라 실제 사용하지 않고 선언만 해둔 빈 메모리 버퍼, 아직 디스크에서 램으로 로드되지 않은 파일 매핑 영역, 그리고 여러 프로세스가 함께 쓰는 공유 라이브러리(libc 등)까지 모조리 가상 주소 공간에 들어와 있다면 전부 덧셈하여 집계.
- 리눅스 커널은 모든 프로세스의 비행 데이터를 가상 파일 시스템인 `/proc` 하위에 실시간 텍스트로 보존
    - `/proc/[PID]/stat`: 프로세스의 상태(Running, Sleeping 등), CPU 시간 정보, 그리고 가상 메모리 및 물리 메모리 페이지 단위 통계가 압축 수치 형태로 담겨 있습니다.
    - `/proc/[PID]/status`: 사람이 읽기 쉬운 포맷으로 VmSize (VSZ)와 VmRSS (RSS)를 나누어 표기해 줍니다.
관제 도구의 내부: 본 미션에서 작성한 monitor.sh 내부의 ps -p "$PID" -o %mem= 명령어 또한 실제로는 커널이 수집해둔 /proc/[PID]/stat 파일 내부의 RSS 페이지 수를 읽어와 시스템 전체 가용 물리 메모리 분모 대비 나눗셈 백분율 연산을 수행하여 우리에게 시각화해 주는 구조로 작동합니다
